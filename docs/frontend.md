# Frontend

A single static shell (`index.html` + `styles.css` + `app.js`) served by the kernel — no framework, no build step. The visual language descends from `prototype/index.html` (the original user-approved reference) and has since evolved under user direction: Lucide icons, bespoke form widgets, fieldset grids. **The chrome wears the HappySkills brand** — a black, fully-rounded topbar "island" and a white card sidebar floating on a lavender shell, a vermilion (`#eb4a26`) app accent, and self-hosted **Inter** for the chrome only; the document pane deliberately keeps the original system font (see [Theming](#theming)).

## Shell

- The token comes from `?token=` (held in memory, never localStorage); asset URLs carry it as a query. `__IC_TOKEN__` placeholders are substituted server-side because CSP forbids inline scripts.
- Hash routing: `#/c/<encoded-rel-path>`. The sidebar is the workspace scan — the `(root)` group displays the workspace folder's real name with a house icon; every folder in the tree holding a renderable file is a collection, at any depth, named by its relative path (the scan is recursive — see [architecture.md](architecture.md)). There is deliberately nothing else to it: no "+" (folders with content appear on their own), no delete (the reader's browser never destroys files), no in-browser workspace switching — all three were removed in 0.8.0 with their kernel routes. **Folders start collapsed** — a recursive workspace can list dozens of collections, and a wall of expanded folders buries the one canvas being read. The group holding the active canvas *derives* open (`groupIsOpen()`), so navigating anywhere — sidebar, search, an agent's `open` — always reveals where the reader is; the reader's explicit clicks (`state.groupOpen`) outrank the derivation both ways, and a fresh navigation into a folder clears its manual collapse, because opening a canvas must reveal it. The topbar shows the workspace path as a **copyable label** (light monospace on the island with a ghost copy button — deliberately *not* the white "search input" look it once had), filling available space and trimming from the *start* (measured fitting + ResizeObserver) so the tail stays visible; the HappySkills wordmark beside it is an `<a>` to happyskills.ai (`target=_blank rel="noopener noreferrer"` — the page URL carries the token, so a referrer would leak it). **The shell is responsive**: below 900 px the sidebar becomes an off-canvas drawer toggled by a hamburger (closes on scrim tap, navigation, or Escape) and the main pane fills the row; below 600 px the island drops the product label and the path, and form fieldset grids collapse to one column. The responsive `@media` rules live at the *end* of `styles.css` so they win the cascade over the base chrome rules (see [gotchas/frontend.md](gotchas/frontend.md)).
- **Canvases and documents share the sidebar and are told apart in it.** A scan entry's `kind` picks its glyph — a canvas keeps its dot, a markdown document wears a `file-text` icon — and both sit in a common 14 px slot so a folder holding each still reads as one left-aligned column of titles. Canvases lead, documents follow. **A document with a COMPANION gets one row, not two**: the companion canvas is dropped from the tree entirely (`scan.js`), because the reader thinks in documents, not in metadata — but the row wears an accent dot (`.enh-dot`, from the entry's `enhanced` field) and its tooltip names the canvas that furnishes it. A state the reader cannot see is a state that teaches nothing. The stat line counts them apart (`3 canvases · 2 docs · 2 groups`). A native document's canvas head drops its title and shows only the file path — the title was lifted from the document's own H1, which the markdown block is about to render anyway.
- `?view=deck` opens any canvas straight into the paper deck. This exists for `print`: a declared `document` decks itself, but a markdown file opens continuous like every other display canvas, so the CLI asks for paper in the URL rather than reaching into the page to click the toggle.
- Hot-reload client: WebSocket with exponential backoff; `workspace` refetches the tree, `canvas` re-renders the open canvas (full re-render, state loss accepted), `navigate` routes, `session` refreshes form state. The footer pulse reflects connection health.
- The sidebar footer's last line is the **running version** (`.side-foot .ver`, from `package.json` via `lib/pkgmeta.js`). It reaches the page as an `__IC_VERSION__` placeholder substituted in `serveShell()`, exactly like `__IC_TOKEN__` — the CSP forbids the inline `<script>` that would otherwise carry it, and fetching one string is wasteful when the shell is already being templated. `render.test.js` asserts the rendered text in a real browser, because an unsubstituted placeholder is invisible to every server-side test.
- **Canvas search** (the sidebar magnifier, `⌘K`, or `/`): a frosted-glass modal — a `backdrop-filter` blur over the `--scrim` token, with an opaque `--panel` panel floating above it. The index needs no fetch and no build step: it is `state.tree`, which the sidebar already holds and the kernel refreshes over the WebSocket, flattened and memoized against the tree object's identity. Matching is token-substring over canvas title, folder, and file name, ranked by a title boost. Result rows are built as **DOM nodes, not an HTML string**, so highlighting cannot leak markup or drop a `<mark>` inside an entity like `&amp;`.

## Display renderers

- **markdown** via vendored markdown-it (`html: false, linkify: true`); `src` content and every workspace-local image arrive **pre-inlined by the kernel** (`data:` URIs — the browser never issues a request, and `img-src 'self' data:` already permits them). Oversize, unreadable, or out-of-workspace images degrade to a labeled fallback, never a broken image. `md.validateLink` is overridden to accept the base64 image types the kernel emits: markdown-it's default allows only png/jpeg/gif/webp `data:` URIs and silently discards the rest, SVG included (see [gotchas/frontend.md](gotchas/frontend.md)). Fenced code is highlighted by vendored **highlight.js** (full build, 192 grammars) through markdown-it's `highlight` hook; only a *declared* language is highlighted, since auto-detection across 192 grammars mislabels short snippets. Two skill-side markdown-it core rules fill gaps the library leaves: GFM **task lists**, and rewriting markdown-it's `style="text-align:…"` column alignment into a **class** (see [gotchas/frontend.md](gotchas/frontend.md)). **Notes and tables carry the palette, not gray.** A blockquote renders as a rounded callout — an accent left-stripe over an `--accent`-tinted surface — and a markdown table gets rounded corners, a tinted header, and zebra body rows. Each tint is a `color-mix()` over `--accent` (never a fixed gray), so one declaration adapts to the app's light/dark theme *and* a document's own accent on paper, without duplicating a value across the three theme blocks (see [Theming](#theming)). Rounded corners require `border-collapse: separate`, with the table's `overflow-x: auto` scroll box doing the corner clipping on screen (see [gotchas/frontend.md](gotchas/frontend.md)).
- **kpi** cards: delta arrow from the value's sign, color green iff sign matches `positiveIs`, near-zero renders flat/muted.
- **table**: per-column `format`, numeric columns right-aligned with tabular numerals.
- **chart**: `chartFigure()` maps each of the 26 kinds' friendly `data`+`encoding` to a Plotly `{data, layout}` figure (bubble scaling, colorscaled heatmaps with cell labels, sankey ribbons tinted by source, hierarchy flattening for treemap/sunburst, `scatterpolar` radar, `indicator` gauges, precomputed-fence boxplots, `pivotGrid()` for surface/contour). Four kinds have no usable Plotly trace and are rendered by the skill: **`graph`** runs a deterministic seeded Fruchterman-Reingold (`forceLayout()`) and draws edges + degree-sized nodes as `scatter`; **`themeRiver`** computes a symmetric streamgraph baseline and draws each band as a closed polygon; **`dendrogram`** turns a linkage into U-bracket polylines (`dendrogramPath()`); **`silhouette`** sorts within each cluster, gaps the groups, and adds a mean reference line. The raw `options` escape hatch is a Plotly figure fragment merged **by trace index** (`applyOptions()`), so a patch refines the generated trace instead of replacing its data.

## Mounting

`mountCharts()` awaits each `Plotly.newPlot` **in sequence** — deterministic order, and a `try`/`catch` per chart so one failure cannot take a neighbour with it (see [gotchas/frontend.md](gotchas/frontend.md)). A generation counter lets a re-render abandon an in-flight mount loop. Each chart gets a ResizeObserver driving `Plotly.Plots.resize`.

**The axis and the legend are fitted after the render, because that is the only moment their geometry exists.** Plotly's automargin treats the tick labels and a legend below the plot as independent pushers of the bottom margin and reserves the **max**, never the sum — so long rotated category labels were painted straight through the legend. The legend is therefore anchored to the **container's** bottom edge (never in paper coordinates, which are a fraction of a plot area that shrinks as the labels grow), and `fitLegendBelow()` measures the lowest tick label against the legend *after* the browser has chosen its tick angle, then relayouts `margin.b` once to hold both bands stacked. It converges because the manual margin is a floor that already exceeds every push. It reads the **DOM, not the block**, so it covers every kind with a horizontal legend; it stands down for sweeps (which tune their own margins for the slider) and for any block whose `options` pins `layout.margin.b` or `layout.legend` — that patch is applied last and is the author's final word. It re-runs on resize and on the deck ⇄ continuous move, since a sheet is narrower than the pane and the ticks re-rotate. `state.fits` counts the relayouts still in flight, and `print` waits on it: a chart has its `.main-svg` one relayout *before* its margin is correct, so a PDF taken on the SVG alone can photograph the deck mid-fit.

**Category tick labels elide at 30 characters** (`catTicks()` on `bar`/`line`/`area`); the x values, the hover and the file keep the whole string. This exists because nothing used to shorten a label, so agents hand-truncated their own data to make it fit — damaging the data to serve a layout they could not see. How much of a label survives on a crowded axis is rendering, and rendering is the runtime's.

Boxes are 320 px by default; `.tall` (460 px) for `scatter3d`/`surface`/`splom`, `.swept` (400 px) when a sweep adds a slider, and both combined (540 px).

## Sweeps

A chart block carrying `sweep` instead of `data` becomes a parameter sweep. `sweepFigures()` builds **one figure per frame** up front, `sweepLayout()` adds a themed Plotly slider (`method: "skip"`, so the step change is a DOM event rather than a Plotly API call — `method: "animate"` is broken upstream for `scatter3d`), and `attachSweep()` listens for `plotly_sliderchange` and swaps the whole figure with `Plotly.react`. The handler lives in `app.js`: **the skill owns the JavaScript, the agent ships only rows.** Because `react` updates in place, dragging a slider across a 3D sweep reuses one WebGL context. `rethemeCharts()` rebuilds every frame on the new palette and holds the reader's current step.

## Document mode: the deck and the packer

A canvas whose envelope carries `document` renders as a **deck of literal page-sized sheets** (`renderDocumentView` in `app.js`). Pagination is by construction: content is packed into fixed-size boxes (`210mm × 297mm` for A4, geometry set through CSSOM), `@media print` resets the screen scale and breaks after each sheet, and a per-canvas `@page { size: …mm …mm; margin: 0 }` rule arrives via a **constructed stylesheet** (CSSOM, exempt from `style-src`). The print engine never chooses a break, so screen and PDF agree — the invariant that carries everything is **`sheet.scrollHeight <= clientHeight`**; a sheet even 3 px too tall silently prints a sliver page.

- **Packer** (`packFragments`): fragments are measured cumulatively inside a hidden auto-height replica sheet at exact content width (the budget comes from a fixed-height probe with the real strips; bodies and fragments are `flow-root` so edge margins cannot escape measurement). Code blocks split by lines via `Range.extractContents` — an hljs span crossing the boundary splits cleanly, DOM surgery, never string surgery; tables split by rows with the `<thead>` repeated; lists by items with `<ol>` numbering continued; paragraphs, charts (fixed-height boxes) and kpi rows are atomic; a heading is never left last on a sheet; an atomic element taller than a page gets its own sheet, clipped (`overflow: hidden` preserves the invariant) with a visible notice. Chapters (`pages[]`) force new sheets.
- **Deck assembly**: the body packs first, then the TOC, so every TOC row can carry the page number its anchor landed on (dotted leaders to the number; digits are written into the already-packed rows — they don't change row height; see canvas-schema.md for the repagination caveat) — then cover + TOC + body + back cover assemble in display order. Header/footer strips are ordinary DOM cloned into every content sheet with `{{pageNumber}}`/`{{totalPages}}` substituted as text after assembly — which is why they survive Cmd+P exactly like the `print` command. On-screen TOC entries scroll to their sheet via `data-doc-anchor` (DOM nodes, never string-built markup).
- **A cover is a sheet, so it can carry a background image.** `applyCoverBackground()` sets `background-image`/`size`/`position` on the **`.sheet` box** through CSSOM — the box, not the padded content box, because a full bleed must reach the paper's edge past the 15 mm margin (`background-clip: border-box`) while the text stays in the padding. CSSOM rather than a style attribute for the usual reason: the CSP drops the latter without a word. The **scrim** is its own absolutely-positioned layer (`.cover-scrim`) between image and text, because a dark photo swallows a near-black title and CSS cannot wash a background in place; **`ink`** repaints the cover's text and *only* the cover's (`theme.text` would paint the whole document, so a white cover title would come with white body text on white paper), and the muted author/date line is derived from it at reduced opacity rather than left to the theme. Z-order: image → scrim → logo/title/subtitle/meta/band. `backCover.background` is the same shape and entirely independent. Neither scrim nor ink is defaulted on — silently tinting somebody's photograph would be presumptuous — so the *validator* warns instead (`COVER_TEXT_MAY_BE_ILLEGIBLE`).
- **Sheets are always light**: `.sheet` redeclares the LIGHT token set, so a dark app renders white paper. Each token the theme can restyle is now wired to its `--doc-*` counterpart with the old literal as the CSS fallback (`--text: var(--doc-text, #1a1d24)`, `background: var(--doc-paper, #fff)`, …), which is what lets an **unthemed** document render byte-identically to before while a themed one repaints. See [Theming](#theming) for the two sinks the theme feeds and what it deliberately does not reach.
- **Paper typography is its own layer, scoped to `.sheet`.** The deck renders each markdown element as its own `.md.doc-frag`, which makes every element both `:first-child` and `:last-child` of an `.md` — so the `.md > :first-child/:last-child` margin reset zeroed the entire document's rhythm, and paper printed with its headings glued to the prose (see [gotchas/frontend.md](gotchas/frontend.md)). The rhythm is re-declared under `.sheet` and loosened for reading on paper (line-height 1.7, generous heading margins), and **code fences wrap** there (`white-space: pre-wrap; overflow-wrap: anywhere`) because a PDF has no scrollbar — an `overflow:auto` fence is not scrollable in print, it is clipped. Wrapping makes fences taller and repaginates the deck; that is safe only because the packer's replica *is* a `.sheet`, so it measures the same wrapped layout the printer produces. **Anything that must apply to markdown on paper belongs under `.sheet`** — that is the one selector both the deck and the measuring replica answer to.
- **A wide table folds instead of losing columns.** The same clip cost a table whole *columns* — an eleven-column table printed with seven and a half — so `tagWideTables()` measures each table in a real sheet body before packing and tags the overflowing ones `.wide`, which switches them to `table-layout: fixed` (the only layout that cannot exceed the measure) with `break-word` cells. Only the offenders: fixed layout divides the page evenly, which would stretch a tidy four-column table across the sheet. Measured with `scrollWidth` — `max-width` clamps the table's box while its columns spill inside it, so `offsetWidth` can never exceed the container and the check would never fire (see [gotchas/frontend.md](gotchas/frontend.md)).
- **No copy button on paper**: `mountCodeCopy(scope, {button: false})` mounts the `.code-block` wrapper without its button, so the deck holds none at all (nothing to hide at print time) and the fence reclaims the right padding the button reserved. The wrapper still goes on *before* the packer runs — a fence's geometry must be settled before it is measured, never after. The continuous view keeps one button per fence.
- **Deck ⇄ continuous toggle** (topbar segmented control, visible for **every** loaded canvas — the view is presentation, not capability). A declared `document` opens as the deck and everything else opens continuous — but that is the *canvas's* default, and **the reader's choice outranks it and is sticky across navigation**. `state.docViewChoice` (null = follow each canvas's own default) records the view the reader clicked, and every canvas opened afterwards honours it: browsing a folder of markdown as paper is one click, not one click per file. A canvas that cannot be decked at all — a form, a confirm, a sweep — falls back to continuous **for that canvas without forgetting the choice**, so the next deckable document comes up as paper again. A display canvas builds its deck **lazily on first toggle** (`renderCanvas` branches on `state.docView`; packing a 20-chart gallery eagerly on every open would be waste). Every deck derives its defaults — the TOC is auto-generated from the document's **headings and chapter names** whenever anything exists to list (`wantToc`; the JSON `toc` key only customizes title/depth). **Block titles are captions, not structure, and are not listed** — interleaving them with the headings printed a numbered outline broken up by unnumbered caption rows. They remain the *fallback* (`tocEntries()`): a canvas with no prose at all has no other structure, so a chart gallery keeps its contents page, A4/15mm, LIGHT palette via `state.docLand` in `palette()`. A topbar contents button (`#tocBtn`) lets the reader toggle the TOC off and on — a repack, since the TOC's own sheets shift every page number after them. A second button (`#stripsBtn`) does the same for the **running header/footer**, which is derived rather than declared: `docStrips()` resolves a three-state reader override (`state.docStrips`: null = auto, on iff the canvas declared strips) into the strips actually rendered, deriving the canvas title and `{{pageNumber}} / {{totalPages}}` when nothing is declared, and letting the reader drop an author's pair just as readily. It repacks for a sharper reason than the TOC does: `packFragments` probes its per-sheet budget from a sheet **with the strips attached**, so they consume content height — turning them on can add a sheet, which moves every page number after it. Both toggles are per-canvas (`renderCanvas` resets them on navigation — deliberately unlike the view choice above, which follows the reader: a TOC and a running header are opinions about *this* document, while deck-or-continuous is an opinion about how one reads) and live in memory only, so `instant-canvas print` — a fresh page load — renders the declared JSON and never sees them; Cmd+P from the reader's own browser does. Interactive canvases keep the toggle with the deck side muted (`.vt-off`, still clickable): the click shows a toast naming the blockers (`deckBlockers`) and why paper refuses them. A **floating print button** bottom-right (`.print-fab`) appears whenever a deck exists (`window.print()` → the native dialog where "Save as PDF" lives; fires `beforeprint`; hidden in print output). Once in document land, both views live in the DOM with one hidden by a class. **Charts exist once** — each view renders an empty `.chart-slot`, and the live plot node moves between slots (`moveChartsTo` + `Plotly.Plots.resize`, never purge + newPlot). Print CSS always shows the deck and hides the continuous view, so Cmd+P prints sheets no matter what is on screen; `beforeprint` relocates the chart nodes into the deck (a `.printing` class keeps the hidden deck laid out off-screen so resize sees real sizes) and `afterprint` restores them. If `beforeprint` ever failed to fire, pagination would still be correct — only chart sizing could degrade.
- **The paper-only controls disable, they do not vanish.** `#tocBtn` and `#stripsBtn` are properties of paper: off the deck there is no sheet to put a header on and no page numbers for a TOC to cite. They used to be `hidden` in the continuous view, which is wrong twice over — the toolbar reflowed as they came and went, moving every other control out from under a cursor already reaching for one, and a reader who never saw them had no way to learn they existed. `paperControl()` in `syncViewToggle()` keeps them in place, sets `disabled`, strips the `active` ring (an "on" ring on a control you cannot press is a lie about the current view) and writes a `title` naming the view they need — with two distinct reasons for the TOC, which is also unpressable *on* paper when the document has no headings to list. This is the rule `#viewDeck` beside them already followed: **a hidden control teaches nothing.** `#paletteBtn` is deliberately **not** in this group (see below).
- **Colors are not a paper-only control, and the continuous view wears the brand.** A theme belongs to the *document*, not to the deck, so `#paletteBtn` stays live wherever a canvas is loaded. That only means anything because the continuous view actually takes the theme: `applyDocumentTheme()` is applied to whichever root is on screen — the deck's `.doc-mode` or the classic render's plain `.canvas`, both of which match `.canvas` — and `palette()` hands charts the document's colorway over the *app* theme's ink. What the continuous view declines is the **paper**: `paper`/`text`/`surface` are sheet tokens, and forcing them on a dark app paints black on black, so only `--accent` and `--doc-link` cross over (`styles.css`, `.doc-mode .doc-html, .canvas:not(.doc-mode)`). The colorway override is gated on a theme having actually been **declared** (`themeIsDeclared()`), because the default colorway is the light one and forcing it on a dark app would hand it back a palette it deliberately does not use. Without any of this, opening the picker off the deck would have given the reader a color picker that visibly does nothing.
- **Save says what it will do, before it does it.** A colour click can now make a *file appear* in the reader's repository — the companion canvas that gives a markdown document an envelope. `GET /api/theme/plan` (fetched when the panel opens, and re-fetched after every save) answers *what would this write touch*, and `#palNote` renders it: **"Save will create README.canvas.json"**, or a warning that a display canvas will gain a `document` and start opening as paper. On a canvas holding a form, a confirm or a sweep, the plan comes back `blocked` and **Save goes disabled with the reason in its `title`** — visible, not hidden, because a control the reader cannot see teaches nothing — while "All documents" beside it still works. That is the honest way out: an interactive canvas cannot carry a `document`, so the workspace default is the only theme it can wear.
- **The palette control** (`#paletteBtn` → `#palettePanel`) is **two screens**, because it does two jobs and conflating them made both illegible. **Pick** (`#palPick`, the default): the preset chips with swatch strips, grouped **Light paper** / **Dark paper** (`#palPresets` / `#palPresetsDark` — a reader who wants a dark deck has already decided, and should not have to read fourteen names they do not want), then the workspace's own palettes, then the document footer — Reset / "All documents" / Save. **Edit** (`#palEdit`): every token, the chart colorway, a name, one Save. A round **+** in the header (`#palAdd`) is the only way in and the whole answer to *"how do I add my own colors?"*; `#palBack` is the way out; `state.palView` says which screen is up and `showPaletteView()` swaps the body. Three rules hold it together, each learned by getting it wrong first: **(1)** the "+" is unconditional, so an empty workspace still advertises the feature — hiding the custom section when there were none meant deleting your last palette deleted every trace that palettes existed. **(2) One screen, one Save.** The document's footer is *hidden* in the editor; leaving it up sounded principled (*the footer always means the document*) and rendered as two identical primary buttons both reading "Save", one writing the document and one the workspace, twelve pixels apart. **(3)** The editor starts from the colors on screen and, if those are one of the reader's own palettes, from its **name** — so saving *updates* it (the button reads `Update`) rather than forking a second near-identical "My brand", and "edit a palette" needs no separate affordance. Saving returns to the list where the new chip sits lit: the chip is the confirmation. **`#palDetail`** sits below the list and outside the scroller — what the selection *is*, so reading it never costs a scroll: name, every token, every chart color, and for a preset the guidance (`description`) that otherwise lives only in a chip's `title` tooltip, where the one thing that would make a reader choose Okabe-Ito over Ember (*colorblind-safe by construction*) was unreadable. `paletteIdentity()` distinguishes three states, and the third is the one the old panel could not express: a preset, one of the reader's own palettes, or **a preset with overrides on top — which is no longer that preset**, and says so instead of leaving its chip lit and staying quiet. Every edit previews live — `previewTheme()` repaints the `--doc-*` properties and calls `rethemeCharts()`, in place, for the same WebGL-context reason the app theme toggle does — and **Save** is what persists it, through `POST /api/theme`. Picking a preset chip *drops* the token overrides, because a chip that does not give you what it shows is a lie. Reset on a canvas that declares its own `document.theme` refuses with a toast rather than editing the author's file (the kernel says the same thing with a 409). Nothing in the page knows what a preset *is*: the kernel hands it hex (see [architecture.md](architecture.md)), and `resolveLocally()` mirrors the layering rules only far enough to preview an unsaved edit before the round trip — but it must mirror them *exactly*, because a preview that disagrees with the kernel is a lie about what Save is going to do. The panel decides whether a click was inside itself in the **capture phase**, which is not an optimization but the only thing that works (see [gotchas/frontend.md](gotchas/frontend.md)).
- **The colorway editor has two subtleties, both consequences of the one-is-a-lead rule** (see [canvas-schema.md](canvas-schema.md)). A swatch can be removed down to a **floor of two**: one color is a *lead* the preset would refill from, so removing the second-to-last would silently undo itself. And the colorway a reader edits is first **materialized** (`currentWay()`) — until they touch it the declared theme may name no palette at all, the preset having supplied one, and an edit to a colorway that is not there cannot be expressed. The first touch pins the whole resolved colorway; every edit after it is exact.
- **Custom palettes are a library, not preset names.** Saving one captures the **resolved** theme, never the declared one (`savePalette()`): a library entry that said only `{preset: "forest"}` would be a pointer rather than a palette, and would repaint under the reader if the preset ever changed. Applying one materializes its colors into the document, so nothing is left in the file to match a chip against by name — `activeCustomName()` therefore matches **by value**, deep-comparing the declared theme against each entry, and a matched custom chip lights up while *no* preset chip does. Deleting an entry (`POST /api/theme/palette` with `theme: null`) repaints nothing: the colors were copied into the documents that use them, and they are still what those documents say. [canvas-schema.md](canvas-schema.md#custom-palettes-a-library-not-a-preset-name) carries why the canvas must not hold the reference. **The reader is not the only author of this library**, which is why `state.themePresets` — fetched once per session, the workspace's palettes riding in it — is **dropped on every `workspace` broadcast**: an agent running `instantcanvas theme --save` writes a palette straight to disk, and a picker still showing the cached list would be showing a library that no longer matches the file it came from.
- **This is deliberately not a third reader toggle.** The TOC and header/footer buttons live in memory and die with the tab, because they are opinions about how to *look* at a document. A theme is part of the document: it must survive a reload, reach `print` (a fresh page load, which never sees a reader toggle), and be readable by the agent that wrote the canvas. Hence a persisted setting with an explicit Save, not a per-tab switch.
- Screen presentation: sheets on the app background with shadows, scaled to fit via one `transform: scale()` on `.deck-scale` (CSSOM; print resets it with `!important`), refit on resize through a ResizeObserver.

## Presentation mode: the filmstrip and the stage

A canvas carrying a `slides` array is a **presentation** — a *sibling* of the document deck, not a variant of it. `renderCanvas` branches on `Array.isArray(canvas.slides)` and hands off to `renderPresentationView` **before any of the deck/continuous machinery** runs, because a deck of slides shares none of it: no packer, no continuous twin, no TOC. `state.presLand` is the flag, mirroring `state.docLand` exactly — and `palette()` hands charts the document colorway over its own surface whenever `docLand || presLand`, so a slide charts like a sheet.

- **The filmstrip is the browse view.** Each slide is a fixed-geometry box — `1280×720px` for 16:9 (`13.333in × 7.5in`) or `960×720px` for 4:3 (`SLIDE_GEO`/`slideGeometry`) — stacked vertically and scaled to fit the pane by **one** `transform: scale()` set through CSSOM (`fitStrip`, the deck's `.deck-scale` pattern; a ResizeObserver refits). Sizing is in **inches**, the same unit as the `@page`, so print is 1:1 with no sub-pixel doubling. The browse chrome sits **below** each box, *unscaled* so it stays readable at any fit: a muted `Slide N of M` label, an overflow badge, and the slide's `notes` paragraph.
- **Seven class-based layout renderers** (`SLIDE_HTML`: `title`, `section`, `content`, `two-column`, `quadrant`, `statement`, `closing`) build each layout's grid, and its regions fill with the very same block renderers the rest of the app uses — `renderMarkdown`/`renderKpi`/`renderTable`, plus a chart-box for charts. A lone chart or KPI **fills** its region (the CSS flexes it; the ResizeObserver `mountCharts` attaches drives `Plotly.Plots.resize`). Backgrounds reuse `applyCoverBackground` on the slide box (image → scrim → content — z-index on a flex item needs no `position`, the `.sheet` cover lesson). The declared running footer is cloned into every slide but title/closing via `slideFooterHtml`, with `{{slideNumber}}`/`{{totalSlides}}` substituted as **text**.
- **A slide is its own surface, like a sheet.** It takes the **full** `--doc-*` token set — paper included — via `applyDocumentTheme` on the `.pres-mode` root, so a dark deck is normal and `data-paper="dark"` swaps the semantic layer. The `.slide` CSS layer in `styles.css` is scoped exactly like `.sheet`, but with its own **large** type scale: a slide is read from meters away, not from a chair.
- **Autofit (D6) measures after the DOM settles**, the same discipline as the packer. `autofitSlides` awaits `document.fonts.ready` first — a badge that measures a half-laid-out slide lies — then `autofitOne(slide)` checks each `.slide-region` for `scrollHeight > clientHeight` and steps the type scale down through **at most three** class-based steps (`.fit-1/2/3` → `--slide-fs`). Still overflowing after that, it adds `.clipped` and reveals a **filmstrip-only** badge — the reader sees the overflow while browsing, and the stage never shows a warning to an audience.
- **The topbar reflects that a deck has no continuous twin (D9).** A `#presentBtn` (**Present**) owns the view-toggle slot in place of the deck⇄continuous pair; `#tocBtn` and `#stripsBtn` **disable with reasons** (a presentation has no table of contents and no running-strip toggle — its footer is declared, not toggled); `#paletteBtn` stays **live** because colors belong to the document; and the print FAB shows. `syncViewToggle` carries a dedicated `presLand` branch for all of this. The sidebar badges a deck with a distinct Lucide `presentation` glyph (`scan.js` sets a `deck` flag; the `kind` stays `canvas`).
- **The presenting stage (Phase D) is a sibling `#stage` overlay** — one slide filling the screen, entered fullscreen via `requestFullscreen()` on the stage, or **in-viewport** when the browser refuses the gesture (D10, never a hard dependency), scaled to contain the viewport by one transform (`fitStage`). `presentStart` opens on the **most-visible** filmstrip slide. The live chart nodes **move** from the filmstrip into the stage slide on entry and back on exit (`stageMoveChartsIn`/`stageReturnCharts` + `Plotly.Plots.resize` — never purge + `newPlot`, so no WebGL context leaks), while each stage slide is built **fresh** (`buildSlide`) and autofit-measured (`autofitOne`). Keyboard control (D11) is scoped to presenting by a **capture-phase** handler that swallows *every* key — so `⌘K` and `/` keep their meaning only elsewhere and do nothing here: →/↓/Space/PageDown/Enter and a click advance; ←/↑/PageUp/Backspace go back; Home/End jump to the ends; typed digits + Enter jump by number; `B` toggles a black overlay; `Esc` exits. The cursor auto-hides after ~2s idle (`.cursor-hidden`) and wakes on move. A native fullscreen exit returns to the filmstrip **at the current slide**; navigating to another canvas leaves presenting; and a hot reload holds the slide index (clamped to the new count) and re-grabs the fresh chart nodes the rebuilt filmstrip just mounted.

## Bespoke form widgets

Everything the user touches is custom — native browser chrome never appears mid-form:

- **date / datetime picker**: calendar popover with month/year quick-select grids (12-year paging), Today/Now/Clear; datetime adds an hh:mm section and a Done button (day-pick keeps it open). The input stays typable ISO, so native `required`/`pattern` validation still applies.
- **select**: styled menu popover, check on the current option, Lucide chevron aligned with the other input icons; the display input is a keyboard-openable trigger (Enter/Space/↓) whose typing is suppressed.
- **radio / checkbox / range**: `appearance: none` rebuilds — accent dot, SVG-check box (data-URI, allowed by `img-src data:`), slider with accent progress fill (`--fill` custom property set from JS).
- **`ui: "buttons"`** segmented control and **`ui: "pills"`** token multi-select (filter input, removable ×) — both store values in hidden inputs so `collectValues()` reads them uniformly.
- **Validation UX**: live on-blur checks (`clientFieldError()` mirrors the kernel's rules — email, URL + protocol whitelist, pattern with `patternMessage`, ranges, dates) render into inline error slots and clear on input. The kernel re-validates on submit regardless; 422 responses render field-level errors, 409 responses drive the overwrite/outside-root confirmation dialogs.

## Layout

`.canvas` has **no max-width**. It is a fixed gutter (`--gutter`, 40px, narrowing to 16px under 720px) and everything else is content — markdown, tables and charts all fill the pane. Tune `--gutter`; do not reintroduce a measure cap.

Fieldsets render as bordered groups (`--fset-border` token) with 1–3-column grids; below 600 px those multi-column grids collapse to one column. **CSP blocks `style=""` attributes**, so all geometry is class-based: `.cols-2/.cols-3` on the grid, `.span-2/.span-3` on fields, utility classes elsewhere. JS may set `el.style.*` (CSSOM is allowed); markup may not carry style attributes.

Every code block rendered **on screen** gets a copy-to-clipboard button. `mountCodeCopy()` runs after mount and wraps each `<pre>` in a `.code-block` positioning context, because the button is chrome rather than document content and must not travel with the markdown. It is **always visible, never hover-gated** (see [gotchas/frontend.md](gotchas/frontend.md)). On paper it is not mounted at all — `{button: false}` puts the wrapper on and nothing else, since nobody copies a PDF to the clipboard. `copyText()` prefers `navigator.clipboard` — `127.0.0.1` is a secure context — and falls back to an off-screen `<textarea>` plus `execCommand`, positioned by the `.offscreen` class rather than a style attribute.

## Theming

**Two different things are called a theme, and they never mix.** The **app theme** is the reader's light/dark preference for the chrome around the content — a per-tab UI opinion. The **document theme** is the brand a *document* carries onto paper: it is resolved by the kernel, persisted to disk, and printed. A sheet is paper in a dark app just as it is in a light one.

The app theme is two mechanisms, deliberately separate:

- CSS custom properties per theme (`:root`, `[data-theme="dark"]`, and the `prefers-color-scheme` fallback), including input tokens (`--inp-*`), fieldset border, syntax-highlighting tokens (`--code-*`, which carry the hljs theme — light mode needs darker hues than the chart palette to stay readable on `--panel-2`), and `color-scheme: light/dark` so native widget internals (number spinners, scrollbars) follow the theme. **All three theme blocks must be kept in step.**
- **Plotly cannot read CSS variables**, so two concrete palettes (`LIGHT`/`DARK`) compile into a `layout.template` via `plotlyTemplate()`. The default colorway is HappySkills orange-led — `#eb4a26 #2e6fd8 #0e9384 #9b51e0 #d6336c`, validated (dataviz six-checks) for colorblind separation and surface contrast on both white and dark paper, so both modes share the one set. The toggle calls `rethemeCharts()`, which rebuilds each figure and applies it with **`Plotly.react` — in place, never `purge` + `newPlot`**: WebGL contexts are never released on teardown, so rebuilding would exhaust the browser's context ceiling (measured: 6 toggles → 6 contexts leaked vs 1 reused). Nothing else needs re-rendering; the CSS variables carry it.

The document theme arrives from the kernel already resolved — every key a literal hex string, no preset names, no layering rules in the browser (`lib/theme.js` owns them; see [canvas-schema.md](canvas-schema.md)). It reaches the page through **exactly the same two sinks, for exactly the same reason**:

- **CSS**, as `--doc-*` custom properties written by `applyDocumentTheme()` onto the `.doc-mode` root through **CSSOM `setProperty`** — the CSP drops `style=""` attributes but exempts programmatic assignment. `.sheet` reads them for its own token set, so everything the sheet already styled with `--text`/`--accent`/`--border` follows for free. That CSSOM route is also precisely *why* strict-hex validation matters: `setProperty` will happily take the string `javascript:alert(1)`, so the value is checked by the validator and re-checked at the `/api/theme` boundary, because the browser is not trusted (colors are only ever handed to `setProperty`, never interpolated into markup).
- **Plotly**, through `documentPalette()`, which compiles the resolved colorway into the figure template — the same "Plotly cannot read `var()`" constraint as above, which is the entire reason a colorway must be concrete hex by the time a chart is drawn rather than a CSS variable like everything else. `palette()` branches on `state.docLand`: in document land a chart paints the document's theme over a light sheet whatever the app theme says.

Two boundaries are load-bearing. **Semantic colors are not themeable** — green/amber/red (a KPI delta means something) and the `--code-*` syntax palette stay put; they carry meaning and are not the author's to rebrand. And **the continuous view takes the brand but not the paper**: `.doc-mode .doc-html` maps `--accent` and the link color and stops there, because it is a screen view that still follows the app's light/dark theme — forcing a document's `paper`/`text` tokens onto it would print black on black.

## CSP shim

`csp-shim.js` loads before `plotly.min.js` and reconciles Plotly with `style-src 'self'`: it plants a `.no-inline-styles` stub so Plotly skips its runtime `<style>` injection (the rules come from the vendored `plotly.css` instead), and reroutes `setAttribute('style', …)` — which the colorbar uses — into CSSOM assignment, which the CSP exempts. Verified in a browser under the real kernel headers: zero violations, zero injected `<style>` elements.

## Icons and vendored assets

All icons are Lucide (ISC) — only the ~20 used SVG paths are inlined (the `LUCIDE` map in `app.js` plus the static topbar), not the library. The **HappySkills wordmark** in the topbar is inlined the same way (an `<img>` could not follow the theme, since a linked SVG cannot see the page's custom properties), with its letter paths set to `fill="currentColor"` so it paints white on the black island; it is wrapped in an `<a>` to happyskills.ai (`rel="noopener noreferrer"`). Provenance for everything third-party lives in `scripts/web/vendor/VENDORED.md`: a **custom strict Plotly.js 3.7.0 build** (no map traces; see the rebuild recipe and why no published dist substitutes), its extracted `plotly.css`, the markdown-it 14.3.0 UMD build, a **full highlight.js 11.11.1** (192 grammars, assembled — no published single file carries them all), and **Inter** (SIL OFL, latin subset, weights 400/500/600/700) for the app chrome — Satoshi, HappySkills' display face, was rejected because its licence restricts redistributing the font *files* and this package ships them, whereas the OFL permits it.
