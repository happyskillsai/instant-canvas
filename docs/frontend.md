# Frontend

A single static shell (`index.html` + `styles.css` + `app.js`) served by the kernel — no framework, no build step. The visual language descends from `prototype/index.html` (the original user-approved reference) and has since evolved under user direction: Lucide icons, bespoke form widgets, fieldset grids. **The chrome wears the HappySkills brand** — a black, fully-rounded topbar "island" and a white card sidebar floating on a lavender shell, a vermilion (`#eb4a26`) app accent, and self-hosted **Inter** for the chrome only; the document pane deliberately keeps the original system font (see [Theming](#theming)).

## Shell

- The token comes from `?token=` (held in memory, never localStorage); asset URLs carry it as a query. `__IC_TOKEN__` placeholders are substituted server-side because CSP forbids inline scripts.
- **Two hash routes.** `#/f/<rel>` opens a folder's **browse view** in the main pane (`#/f/` = the workspace root, where the app lands); `#/c/<rel>` opens a canvas, a markdown document, or an image. `route()` parses both, and `renderCanvas` branches to the browse view when `state.browseId` is a string. Browse tiles navigate items to `#/c/` and folders to `#/f/`; the `#/c/` route opens its item in a **frosted-glass modal** over the browse view (below). The topbar shows the workspace path as a **breadcrumb** (`#rootpath`, light monospace on the island): the segments come **server-computed** in `state.tree.crumb` (every ancestor of the root, filesystem-root → current, each `{name, path, current, clickable}`), so the browser needs no path/casing/separator logic. An ancestor that is **at or below `$HOME`** renders as a **`.rp-link` button that re-roots the workspace up to it** — `POST /api/workspace/open`, then `location.href` to the returned URL (a *different* kernel; new port + token). Segments above home and the current folder are plain text. The row is a horizontal scroller (scrollbar hidden) kept scrolled to the **end** so the current folder shows, with higher ancestors reachable by scrolling. The **copy icon** (`#rootpathCopy`) still copies the full path — the breadcrumb itself no longer copies, since its clicks navigate. **Below 600 px the breadcrumb is hidden** (no room), so the sidebar's root (house) row carries a **`Move workspace` pencil** (`.ws-edit`, shown only at that width) that opens a modal (`openReRootDialog`) listing the current folder and the **tappable** parent folders up to `$HOME` — the same server-computed `state.tree.crumb` and the same guarded `reRootWorkspace()`, with no path to type. So there is exactly one re-root affordance at every size: the topbar breadcrumb at ≥600 px, the sidebar pencil below it. The HappySkills wordmark beside the path is an `<a>` to happyskills.ai (`rel="noopener noreferrer"` — the page URL carries the token, so a referrer would leak it). **The shell is responsive**: below 900 px the sidebar becomes an off-canvas drawer toggled by a hamburger (closes on scrim tap, navigation, or Escape) and the main pane fills the row, while at ≥900 px that same hamburger **collapses/expands** the static sidebar (a `width`/`flex-basis` transition on `.sidebar`; the main pane's `flex:1` fills the freed space in the same motion, `body.sidebar-collapsed`); below 600 px the island drops the product label and the path, and form fieldset grids collapse to one column. The responsive `@media` rules live at the *end* of `styles.css` so they win the cascade over the base chrome rules (see [gotchas/frontend.md](gotchas/frontend.md)).
- **The sidebar is a folders-only tree** (`buildTree`/`openInto`/`syncTreeActive` in `app.js`), fed lazily by `GET /api/dir?…&dirs=1` per expanded level (cached in `state.dirChildren`) — no file leaves; the files live in the browse view. The root row is the workspace folder's real name with a house icon; every other row is a folder with a chevron (its own **single-click** hit target — a hidden double-click descend is undiscoverable, the folder-browser postmortem in [gotchas/frontend.md](gotchas/frontend.md)) and the name, whose click navigates to that folder's `#/f/`. **Dot-folders render like any other folder** — the genuinely noisy ones (`.git`, `node_modules`) are excluded from the listing entirely, so a *shown* dot-folder is a real project folder and graying it only misleads (the API still flags it `hidden: true`; the client no longer styles it differently). `.git`/`node_modules` never appear (the kernel omits them). Expansion and the active highlight are **incremental** — an inserted subtree and a class toggle, never a rebuild — because a list that rebuilds under a gesture loses the node the gesture is on; a full `buildTree` runs only on boot and on a `workspace` broadcast. A folder derives open iff it is a strict ancestor of the active folder (the old `groupIsOpen` rule, on a nested tree); the reader's explicit expand/collapse (`state.treeOpen`) outranks it, and navigating in clears a manual collapse on the path. The footer keeps the stat line (`3 canvases · 2 docs · 4 groups`, still from `/api/workspace`) and version.
- **The browse view** (`renderBrowse`, in `#mainView`) is a mixed grid/list of one folder's renderable items — a *second* navigation surface alongside the tree. A **breadcrumb** of the folder path leads the pane (`.browse-crumb`, reusing the modal's `.oc-seg`/`.oc-slash`): a house to the workspace root, then one clickable segment per folder, the current folder last and non-navigating — the pane's way *up*, since (unlike the modal) it has no ×. It generalises the gallery grid to mixed types: the root carries `.gallery` too, so it reuses the toolbar/tile/selection/list/delete styles, and it follows the same "sort MOVES nodes, live refresh DIFFS by path, selection is a class toggle" discipline — to which the type filter adds a fourth: **a TYPE change HIDES (folder scope) or DIFFS (subtree scope), it never rebuilds**. A *scope* or *layout* change still rebuilds, because those genuinely change what a tile is. Items are **grouped, and child folders lead**: folders → canvases → env → documents → images → videos → audios (a `.env`/`.env.*` is its own `env` kind, opening the edit form); the group order is fixed and the chosen sort (name/created/size + direction) applies WITHIN each group. **A single toolbar `Filter` button opens the filter — everything else lives in a modal**, so the toolbar stays `count · sort · Filter · Select · grid/list` rather than sprouting a whole second bar (on a phone the sort control drops to a full-width second row via flex `order`, leaving Filter · Select · grid/list on the first). The button carries an **active ring + a count badge** (`.g-filter.on` / `.g-badge`) so an applied filter is visible while the modal is closed. The modal (`openFilterDialog`, a `.g-modal` frosted card) holds a **Type** section — chips for **Folders**, Canvases, **Env**, Docs, Images, Videos, Audio, and a wide **Media** row that toggles the three media kinds together and, while on, shows the trio **included and locked** (`.is-included`) — a **Scope** segmented control (**This folder** / **All subfolders**) with a one-line hint, a **live result line** (`N matches · all subfolders`), and **Reset / Done**. Everything applies **live** — toggling repaints the grid behind AND the modal (`filterRepaint`), so the count moves as you go. State is sticky across folders like sort/layout (`state.browseTypes` / `state.browseScope`). **Folders is its own type** (`typeOK`): folders show only in folder scope, and only when *nothing* is selected (All) or `Folders` is chosen — so filtering to Images now hides the folders too. **Folder scope filters the loaded listing on the CLIENT** — instant, no refetch: the listing already holds every kind, so `buildAll` mounts a tile for **every** item and the filter is pure visibility (`applyTypeVisibility` sets `hidden` per `typeOK`, the one code path that decides what shows). Nothing is destroyed, so an `<img>` keeps its decoded bytes across a chip toggle, and the toggle re-orders nothing — the mounted order never reads the type set. A hidden tile is `display: none` through **its own explicit `[hidden]` rule per layout** (an author `display` outranks the UA's — see [gotchas/frontend.md](gotchas/frontend.md)), so its lazy `<img>` never intersects the viewport and never fetches: filtering to Canvases in an image-heavy folder actively *stops* image loading. **Subtree scope refetches** `/api/dir?recursive=1&types=…` so the server does the recursive, *before-the-cap* kind filter (see [architecture.md](architecture.md); the client-only `folder` kind is never sent), drops the folder tiles, and gives each flattened item a **clickable path caption** (`.bt-path` — the folder it lives in, navigating to `#/f/<dir>`, decided in the delegated click so it survives a live sync) — the whole point of the scope: spot a file visually anywhere in the tree, then jump to where it is. Counts, truncation, the empty state, and the overlay's prev/next order all read the **shown** (post-filter) set; every control is class-based (`document.querySelectorAll('.browse [style]').length === 0` still holds, modal included). Every tile is class-based (`document.querySelectorAll('.browse [style]').length === 0` is asserted): a **folder** tile is the folder glyph in a warm accent chip with a `FOLDER` kicker and no file name, opening `#/f/`; a **canvas / env / document** tile is an editorial card — an icon chip (top-left) and kind kicker anchored at the top, then a bold 2-line title and the actual file name in muted mono (a companion accent dot for an enhanced document; an `env` tile a **lock** glyph and `Env` kicker), never a thumbnail (§ no-preview); an **image** tile is a thumbnail (`/api/gallery/file?path=…&v=<mtimeMs>`) or a placeholder card for a non-renderable HEIC/TIFF; a **video** tile shows a **client-captured first-frame poster** with a duration badge (an off-DOM `<video>` drawn to a canvas, `toDataURL` — same-origin so untainted; at most two captures at once in a plain queue, each element released after use, a failure leaving the film placeholder — value-synced onto the existing tile, never a rebuild). That capture is **gated on the filter showing the tile**, and deferred to the first reveal otherwise: an off-DOM `<video>` is not reachable by its tile's `display: none`, so an ungated capture would fetch and decode frames for videos nobody can see (measured: six hidden tiles, six requests, and none left to make once revealed). An **audio** tile is a music placeholder card. **Every tile carries `content-visibility: auto`** (`.gt`, so the authored gallery block gets it too), which lets the browser skip layout and paint for tiles parked far off screen — a 2000-cap subtree listing used to rasterize the whole grid on any repaint. It sits on the tile, never on the `.g-tiles` container, which would skip the entire grid; `contain-intrinsic-size` carries the `auto` keyword so the browser remembers each tile's real size after first paint, with an honest 54 px estimate for list rows. The `@media print` block sets it back to `visible`: relevancy is viewport-based and a printed page has no viewport. The count line reads `N folders · N canvases · N docs · N images`, inserts ` · N env files` after canvases when any exist, and appends ` · N videos` / ` · N audio files` when non-zero (`KIND_WORDS` carries the `env` label). **Live refresh** rides the `workspace` broadcast (refetch `/api/dir`, diff by path; a media tile's changed mtime re-captures its poster). **Selection now spans EVERY renderable kind** — canvases, documents, images, video and audio — because the selection is a *record an agent acts on*, not an in-browser delete. Select mode (button, ~500 ms long-press, Cmd/Ctrl-click) toggles the `selected` class on those tiles (a **folder** stays non-selectable navigation; a canvas/document card carries the same check overlay as a media tile, nudged top-right off its icon chip). The selection is a **workspace UNION** (`state.selection`, a `Map<rel → kind>`) that is **not** cleared on folder navigation — select in folder A, navigate to B, select more, both stay selected — **persisted** on every change to `POST /api/selection` (debounced, the whole set) and **restored** from `GET /api/selection` on boot and each `workspace` broadcast (`restoreSelection` mutates the Map in place and re-applies the class), so a reload, a kernel restart, or a CLI `selection --clear` all reflect in the open browser; select mode itself is sticky (`state.selecting`). **`Select` is the mode toggle** — lit (`.g-select.on`) while active; clicking it **exits** select mode and **keeps** the selection (there is no separate `Done` button, which read as redundant with Clear). Only the explicit **Clear** empties the union (and records the empty set). The `Select` affordance appears whenever the folder shows **any** selectable item (a canvas or document counts now, not just media). **The delete invariant is preserved:** the media-delete button **filters the selection to the media subset before posting** to `POST /api/gallery/delete` (which fails the whole batch on any non-media path — `NOT_A_MEDIA_FILE`), its count-exact confirm counts only those media items ("a count in a confirmation is a promise"), and it disables at zero media — so a canvas or document path **never** reaches the delete route. The reader's browser gains *selection* of canvases and documents but not *deletion* of them; acting on the recorded set is the agent's job (see [architecture.md](architecture.md) and [security.md](security.md)). The authored `gallery` block keeps its own recursive grid and its detail modal — whose zoom/pan stage is now the shared `createImageStage()` (see the overlay's image renderer below).
- **The item modal.** `#/c/<rel>` opens a canvas, document, image or presentation in a large **frosted-glass modal** (`#docModal` — a scrim + a `~94vw/94vh` card, full-bleed on mobile) OVER the owning folder's **browse view, which stays rendered behind it** (`renderPane` keeps `#mainView` on the pane folder and only re-renders when the folder changes; a cold deep link — a shared URL, or `print` — has nothing behind, so the scrim is just a plain frosted backdrop). It *presents* like a modal but **is a route**, not a dismissible popup: the frosted scrim is **decorative — no click-to-close**, so a credentials form is never dropped by a stray click, and **Esc / ×** navigate to `#/f/<dirname>`. The card holds the chrome bar — a **breadcrumb** back to the owning folder, sibling **prev/next** across all kinds in browse-view displayed order (`browseSorted`, recorded by the browse view or derived from `/api/dir` for a cold link), and the **action cluster relocated from the topbar island** (view toggle, Present, TOC, strips, colors — the *same nodes* moved at boot so `syncViewToggle`'s disable-with-reason and the palette panel's capture-phase rules keep working; the island keeps only theme + stop) — plus `#docModalView` (the scrolling content), the **info drawer** (below) and the floating print button. Esc/←/→ are inert while another surface owns the keyboard: the presenting stage (which **stacks above the modal** at z-index 200 and, on exit, returns to it rather than the folder), `⌘K`, the palette panel, an open **folder context menu**, a gallery-block modal or its selection mode, the mobile nav drawer, or focus inside a form. **Print is the load-bearing invariant** the modal must not break: the deck lives inside the modal, so `@media print` hides the whole `.app` (chrome + pane behind) and resets `.doc-modal`/scrim/card/view to a **static, full-page, chrome-free** container — every scrim, width, transform and border stripped — or the deck paginates at the wrong geometry and prints sliver pages. Verified by a printed PDF's `/Count`, never by the CSS.
- **The item info drawer** (`#docInfoDrawer`) is the single home for a file's metadata for **every** kind the modal opens — canvas, document, image, video, audio. An **info button** (`#ocInfo`, the Lucide `info` glyph) is the **last** control in the chrome bar; it toggles a right-side, **collapsed-by-default** drawer that overlays `#docModalView` (absolutely positioned inside `#docModalCard`, so opening it never reflows the content). `renderCanvas` resets it **collapsed and repopulates it on every open, including prev/next** (a settled no-stickiness decision), and reveals `#ocInfo` for every kind but a presentation. `#ocInfo` / `#infoClose` toggle it, and Esc collapses it **first** when open — before the media-popover, fullscreen and `ocClose` escapes — without navigating (`location.hash` untouched), while ←/→ still flip siblings (the drawer is deliberately *not* in the keyboard-yield list above); `i` toggles it too. Everything renders into one `#docInfoPanel` (`.g-meta`): media flows through `renderMeta` from the stage (which now takes a caller-provided panel, so Duration/Dimensions **value-sync into the drawer even while it is collapsed**), and canvas/document through `renderItemMeta` from `GET /api/meta` — universal `stat` rows (Name/Folder/Path/Kind/Size/Created/Modified) plus **best-effort** extras (Created with, Theme, Blocks, and a document's *Enhanced by* companion), each omitted cleanly when its source is absent. Reuses the shared click-to-copy rows below; all layout is class-based (CSP), it carries its **own** `[hidden]` rules (an author `display` beats UA `[hidden]` — see [gotchas/frontend.md](gotchas/frontend.md)), collapses to a **full-width sheet ≤600px** (that `@media` rule sits last), and is `display:none` in print so it never perturbs deck pagination.
- **Images render in the overlay too** (`#/c/<image>`). `renderCanvas` classifies the routed path by extension against the image union set — templated into `<body data-image-exts>` from `lib/gallery.js` (`GALLERY_IMAGE_EXTS`), so the browser reuses the shared predicate with **no copied list**, and `/api/canvas` is never called for an image. The zoom/pan stage (image on a `--panel-2` stage, wheel/button/dbl-click zoom, drag-pan via one CSSOM transform, metadata from `/api/gallery/meta` with the `?v=<mtime>` cache-buster) is `createImageStage(metaPanel)`, **shared** with the gallery block's detail modal (which wraps it in its own close/prev/next chrome, scoped to the block's images). The metadata panel is now **caller-provided**: the item modal passes the shared info-drawer panel (`#docInfoPanel`), so the meta lives in the drawer and the stage holds no `.g-meta`; the gallery block's modal passes none and keeps its own in-stage panel (unchanged). A non-renderable HEIC/TIFF shows the metadata card, never a broken `<img>`. `syncViewToggle` gains an image branch: the deck/continuous toggle and Present hide (no image analog), and TOC/strips/colors disable **with a reason** — an image is not a document. `body.image-overlay` makes the modal view a flex box so the stage fills it; prev/next crosses freely between images and documents.
- **Video and audio render in the overlay as a fully bespoke player** (`#/c/<video|audio>`, `createMediaStage(kind, metaPanel)`). `renderCanvas` classifies the routed path against the video/audio unions (`isVideoPath`/`isAudioPath`, templated into `<body data-video-exts data-audio-exts>` from `lib/gallery.js` — no copied list), and `/api/canvas` is never called. The player never shows the browser's native `controls` (D3): a `<video>` sits on a `--panel-2` stage (a bare `<audio>` is never shown — an art card stands in), and a transport bar carries play/pause, an `m:ss / m:ss` readout, a scrubber, mute + volume, a **0.5×–3× speed popover** (the select-menu pattern; `state.mediaRate`, re-applied on every mount so speed is sticky across items and across video↔audio), and fullscreen for video. Scrubber/volume are the bespoke `<input type=range>` widget with `setRangeFill`; the scrubber is **skipped by the `timeupdate` sync while the reader holds it**. Duration and (video) Dimensions **value-sync** into the drawer's meta panel on `loadedmetadata` — reaching it even while the drawer is collapsed, since the panel is the caller-provided `#docInfoPanel` — and the server ships null dims, so there is no server-side media parsing. An element error or a metadata-only kind (`.mov`) swaps in a placeholder card, never a dead player. **`dispose()` is load-bearing** (`pause` + `removeAttribute('src')` + `load()`): a detached media element keeps playing in Chrome until GC, so `renderCanvas` disposes the outgoing stage before mounting the next (see [gotchas/frontend.md](gotchas/frontend.md)). Keyboard (D8): Space play/pause, ←/→ seek ±5s (prev/next stays on the ‹ › buttons), m mute, f fullscreen; the speed popover eats Esc, and Esc yields to the browser while fullscreen. `syncViewToggle` gains a media branch (deck controls hidden; TOC/strips/colors disabled with a reason), and `body.media-overlay` makes the modal view a flex box like the image overlay.
- **Every metadata-panel row is click-to-copy** (D6, images included). The shared `metaRow`/`renderMeta` (the info drawer for every item-modal kind, the gallery block's detail modal, and the media player) put an **always-visible** copy icon on each row — never hover-gated, which a touch screen cannot reach — that copies the row's displayed value (the Path row keeps the absolute path), flashes a tick, and toasts. `renderItemMeta` is the sibling that renders a **canvas/document** into the same rows from the `/api/meta` stat plus best-effort provenance. `render.test.js`-style resting-visibility is asserted.
- `?view=deck` opens any canvas straight into the paper deck. This exists for `print`: a declared `document` decks itself, but a markdown file opens continuous like every other display canvas, so the CLI asks for paper in the URL rather than reaching into the page to click the toggle.
- **The tab title names the PDF.** `<title>` ships static as `InstantCanvas`, but `renderCanvas()` overwrites `document.title` per canvas via `pdfDocTitle()` — the cover title, else the envelope `title`, else a markdown file's H1, slugified (lowercase, whitespace → dash, non-alphanumerics stripped), with a full-timestamp generic fallback (`2026-07-17-1430-instant-canvas`) when nothing is usable; it resets to `InstantCanvas` on the browse view. This is what Chrome's `Page.printToPDF` writes as the PDF's `/Title` and what Cmd+P suggests as the filename, so `print` and a reader's own print get the same name (see [cli.md](cli.md)).
- Hot-reload client: WebSocket with exponential backoff; `workspace` rebuilds the folder tree and refreshes an open browse view (both preserving reader state — expansions, selection), `canvas` re-renders the open canvas (full re-render, state loss accepted), `navigate` routes by `kind` (dir → `#/f/`, file → `#/c/`), `session` refreshes form state. The footer pulse reflects connection health — and tells two deaths apart. A dropped socket against a live kernel reads `reconnecting` and heals itself; three straight `/healthz` misses (tokenless by design, so a page that has lost everything else can still ask the old port) read `disconnected` and reveal the footer's **Reconnect** button. Its dialog is honest about the limit — a browser page cannot restart its own local server — and hands over the exact restart command (`cd` into the workspace, then `npx …@latest open .` — the cwd is the CLI's workspace, and `@latest` defeats both npx's sticky cache and the local-project short-circuit inside this very repo) beside an always-visible copy button; the command is OS-aware (a drive-letter root means Windows, read off the path — the kernel's own testimony — never `navigator`; Windows gets a two-line form because no one-line joiner suits both cmd and every PowerShell) and **wraps** rather than scrolls, so a long workspace path cannot hide its own tail. The **stop button's** "kernel stopped" pane shows the same command through the same `commandRow()` — it replaces the whole body, which is why every footer touch in the WS/probe handlers is guarded — and is a terminal state: ANY kernel answering after a deliberate stop reloads the page, without waiting for three misses (a quick restart would otherwise reconnect the WebSocket underneath the pane and leave it up forever). The page keeps probing, and when a kernel answers again it reloads itself: a respawned kernel reuses the workspace identity (same port + token — see [architecture.md](architecture.md)), and a light tokened `/api/dir` check first proves the responder is *our* kernel rather than another workspace's landed on the port. One miss is never a death — the same tolerance the CLI's session poll learned ([gotchas/runtime.md](gotchas/runtime.md)).
- The sidebar footer's last line is the **running version** (`.side-foot .ver`, from `package.json` via `lib/pkgmeta.js`). It reaches the page as an `__IC_VERSION__` placeholder substituted in `serveShell()`, exactly like `__IC_TOKEN__` — the CSP forbids the inline `<script>` that would otherwise carry it, and fetching one string is wasteful when the shell is already being templated. `render.test.js` asserts the rendered text in a real browser, because an unsubstituted placeholder is invisible to every server-side test.
- **The folder context menu is one shared component** (`openContextMenu()` in `app.js`, `.ic-menu` in `styles.css`), because two menus that can disagree about what an action does are two different products. Four items for any folder: **Open in Finder / Show in Explorer / Open in file manager** (the label comes from `<body data-platform>` — the kernel's testimony, never `navigator`; see [architecture.md](architecture.md)), **Open in terminal**, then **Copy path** and **Copy name**. The first two `POST /api/reveal`; every outcome ends in a `toast()`, including `{ok:false, code:'NO_FILE_MANAGER'}`, so an action is never a silent no-op. **Four anchors, all delegated**: a sidebar tree row (from `#tree`, which outlives every rebuild and every incremental subtree insert), a browse **folder tile** and the browse **breadcrumb** (from the browse root and the crumb node — the browse view syncs tiles *in place* on a `workspace` broadcast, so a listener bound to a tile is not guaranteed to survive), and an always-visible **⋮ button** last in the browse toolbar, which acts on the current folder and is the discoverability partner for right-click. It wears the existing `.menu` / `.menu-item` skin (the select widget's and the media speed popover's) and overrides only geometry, exactly as `.m-rate-menu` does — `.ic-menu` is `position: fixed`, because it is appended to `<body>` and placed from a pointer event's *viewport* coordinates, and it clamps to flip left or up rather than render half off-screen. Items are built as **DOM nodes, never an HTML string** (a folder name is user data), left/top are written through **CSSOM** (the CSP drops `style=""` attributes but exempts programmatic assignment), and `e.preventDefault()` fires on **these four targets only** — this is a developer tool and Inspect Element must keep working over prose, charts and tiles. The open menu joins the overlay's keyboard-yield list, so Esc closes the *menu* rather than also navigating to the parent folder.
- **Canvas search** (the sidebar magnifier, `⌘K`, or `/`): a frosted-glass modal — a `backdrop-filter` blur over the `--scrim` token, with an opaque `--panel` panel floating above it. The index needs no fetch and no build step: it is `state.tree` (the scan, still fetched from `/api/workspace` and refreshed over the WebSocket even though the sidebar itself is now a folders-only tree), flattened and memoized against the tree object's identity. Matching is token-substring over canvas title, folder, and file name, ranked by a title boost. Result rows are built as **DOM nodes, not an HTML string**, so highlighting cannot leak markup or drop a `<mark>` inside an entity like `&amp;`.

## Display renderers

- **markdown** via vendored markdown-it (`html: false, linkify: true`); `src` content and every workspace-local image arrive **pre-inlined by the kernel** (`data:` URIs — the browser never issues a request, and `img-src 'self' data:` already permits them). Oversize, unreadable, or out-of-workspace images degrade to a labeled fallback, never a broken image. `md.validateLink` is overridden to accept the base64 image types the kernel emits: markdown-it's default allows only png/jpeg/gif/webp `data:` URIs and silently discards the rest, SVG included (see [gotchas/frontend.md](gotchas/frontend.md)). Fenced code is highlighted by vendored **highlight.js** (full build, 192 grammars) through markdown-it's `highlight` hook; only a *declared* language is highlighted, since auto-detection across 192 grammars mislabels short snippets. Three skill-side markdown-it core rules fill gaps the library leaves: GFM **task lists**, rewriting markdown-it's `style="text-align:…"` column alignment into a **class**, and expanding server-rendered **math** into inline SVG (see [gotchas/frontend.md](gotchas/frontend.md)). **Math is rendered server-side, not in the browser.** LaTeX written as `$…$`/`\(…\)` (inline) or `$$…$$`/`\[…\]` (display) is typeset to `<svg>` by the kernel (`lib/mathsvg.js`, vendored MathJax — see [canvas-schema.md](canvas-schema.md)) and arrives inside the markdown text as an inert PUA+base64 **sentinel**; the `math` core rule decodes it and injects the `<svg>` as inline HTML — inline, so it inherits `currentColor` and themes for free, where an `<img>`-linked SVG could not. Baseline is set by an `mv-<bucket>` class (never an inline `style=""` the CSP would drop), display math centers in a `.math-block`, and bad LaTeX becomes a tinted `.math-error` showing the source. **Notes and tables carry the palette, not gray.** A blockquote renders as a rounded callout — an accent left-stripe over an `--accent`-tinted surface — and a markdown table gets rounded corners, a tinted header, and zebra body rows. Each tint is a `color-mix()` over `--accent` (never a fixed gray), so one declaration adapts to the app's light/dark theme *and* a document's own accent on paper, without duplicating a value across the three theme blocks (see [Theming](#theming)). Rounded corners require `border-collapse: separate`, with the table's `overflow-x: auto` scroll box doing the corner clipping on screen (see [gotchas/frontend.md](gotchas/frontend.md)).
- **kpi** cards: delta arrow from the value's sign, color green iff sign matches `positiveIs`, near-zero renders flat/muted. Each value is a single line whose font shrinks to fit its card (`fitKpiRow` measures `scrollWidth` vs `clientWidth` after mount and on resize, and scales the **whole row** uniformly through one inherited `--kpi-fit` — so a long `US$16,800,000` and a short `1.2%` stay the same size, and a value never clips, in any view). Same skin for **table** blocks as markdown tables — a tinted header and zebra body rows, color-mixed over `--accent` so they follow the theme.
- **table**: per-column `format`, numeric columns right-aligned with tabular numerals.
- **chart**: `chartFigure()` maps each of the 26 kinds' friendly `data`+`encoding` to a Plotly `{data, layout}` figure (bubble scaling, colorscaled heatmaps with cell labels, sankey ribbons tinted by source, hierarchy flattening for treemap/sunburst, `scatterpolar` radar, `indicator` gauges, precomputed-fence boxplots, `pivotGrid()` for surface/contour). Four kinds have no usable Plotly trace and are rendered by the skill: **`graph`** runs a deterministic seeded Fruchterman-Reingold (`forceLayout()`) and draws edges + degree-sized nodes as `scatter`; **`themeRiver`** computes a symmetric streamgraph baseline and draws each band as a closed polygon; **`dendrogram`** turns a linkage into U-bracket polylines (`dendrogramPath()`); **`silhouette`** sorts within each cluster, gaps the groups, and adds a mean reference line. The raw `options` escape hatch is a Plotly figure fragment merged **by trace index** (`applyOptions()`), so a patch refines the generated trace instead of replacing its data.
- **gallery** mounts like a chart — the block renders an empty `.gallery` shell and `mountGalleries()` populates it after the DOM lands (`createGallery()` per block, tracked in `galleryInstances`, disposed on every re-render). It fetches `GET /api/gallery` and builds a grid of `aspect-ratio: 1` tiles (`<img loading="lazy">` at `/api/gallery/file?path=…&v=<mtimeMs>&token=…`, `object-fit: cover`) or a list of rows; a non-renderable HEIC/TIFF gets a placeholder card, never a broken `<img>`. **Sorting re-orders the existing tile nodes** (move, not rebuild) so a selection and the modal survive it. **Live refresh rides the `workspace` broadcast** (`refreshGalleries()`): the listing is diffed **by path** and synced in place — new tiles inserted at their sorted position, vanished ones removed, a changed mtime bumping the `?v` — never a wholesale rebuild, because the DOM is not a pure function of state while a selection or the modal holds references into it (the structural-render vs value-sync rule, [gotchas/frontend.md](gotchas/frontend.md)). Selection is a **class toggle + a `Set` of paths** (entered via a `Select` button, a ~500 ms long-press, or Cmd/Ctrl-click), never a grid re-render; bulk delete goes through a count-exact confirmation dialog to `POST /api/gallery/delete`. The **detail modal** puts the image on a `--panel-2` stage with wheel/button zoom and drag-pan (one CSSOM `transform`), prev/next over the current sort order, and a metadata panel (dimensions from `/api/gallery/meta`); it locks `.main`, decides inside/outside clicks in the **capture phase**, and restores focus to the opening tile on close. All layout is class-based — `document.querySelectorAll('.gallery [style]').length === 0` is asserted in the browser. The toolbar (count, sort control, grid/list toggle, Select) shares one control height and stacks into tidy rows on a narrow screen.

## Mounting

`mountCharts()` awaits each `Plotly.newPlot` **in sequence** — deterministic order, and a `try`/`catch` per chart so one failure cannot take a neighbour with it (see [gotchas/frontend.md](gotchas/frontend.md)). A generation counter lets a re-render abandon an in-flight mount loop. Each chart gets a ResizeObserver driving `Plotly.Plots.resize`.

**The axis and the legend are fitted after the render, because that is the only moment their geometry exists.** Plotly's automargin treats the tick labels and a legend below the plot as independent pushers of the bottom margin and reserves the **max**, never the sum — so long rotated category labels were painted straight through the legend. The legend is therefore anchored to the **container's** bottom edge (never in paper coordinates, which are a fraction of a plot area that shrinks as the labels grow), and `fitLegendBelow()` measures the lowest tick label against the legend *after* the browser has chosen its tick angle, then relayouts `margin.b` once to hold both bands stacked. It converges because the manual margin is a floor that already exceeds every push. It reads the **DOM, not the block**, so it covers every kind with a horizontal legend; it stands down for sweeps (which tune their own margins for the slider) and for any block whose `options` pins `layout.margin.b` or `layout.legend` — that patch is applied last and is the author's final word. It re-runs on resize and on the deck ⇄ continuous move, since a sheet is narrower than the pane and the ticks re-rotate. `state.fits` counts the relayouts still in flight, and `print` waits on it: a chart has its `.main-svg` one relayout *before* its margin is correct, so a PDF taken on the SVG alone can photograph the deck mid-fit.

**Category tick labels elide at 30 characters** (`catTicks()` on `bar`/`line`/`area`, and `dendrogram`'s leaves through the same `shortTick`); the x values, the hover and the file keep the whole string. This exists because nothing used to shorten a label, so agents hand-truncated their own data to make it fit — damaging the data to serve a layout they could not see. How much of a label survives on a crowded axis is rendering, and rendering is the runtime's.

**How MANY labels survive is the same question one axis wider, and it had the opposite answer.** `catTicks` says `tickmode: 'array'` — the only way to hand Plotly its `ticktext`, and *also* exactly what turns Plotly's own tick thinning off. So every kind routed through it demanded one label per row: a 731-day line printed 731 ticks into a 568 px axis (0.8 px each, a solid black smear) while a `candlestick` beside it, passing no tickvals, let Plotly thin 400 dates to a comfortable 40. There are two answers, because there are two kinds of x:

- **A date axis is continuous.** `isDateAxis()` (against `DATE_RE`, single-sourced from the validator — see [architecture.md](architecture.md)) sends `line`/`area`/`bar`/`candlestick` to `xaxis.type: 'date'` with no tickvals at all, and Plotly picks month/quarter/year boundaries that are right at every width. `errorBars`, which never set an axis type, had been doing this correctly the whole time — 200 dates, 7 ticks.
- **A category axis keeps its array** (the ticktext has nowhere else to live) and the runtime thins it: `fitAxisTicks()` keeps every k-th label, k being the smallest stride that leaves each survivor `TICK_MIN_PX` (12 px, the number the validator's warnings quote). It always thins from the arrays the figure was *built* with (stashed by `captureTicks`) — thinning an already-thinned axis would take every k-th of every k-th on each resize and walk the axis down to one label. Measured: 60 categories → 60 at 1246 px, 30 at 652 px, 30 at 456 px, and back to 60 when the pane grows again.

**Heatmap cell labels are a geometry decision too** (`fitHeatmapLabels()`). The gate used to be `rows.length <= 120` — a row count standing in for a question about cell size — so a 144-row grid with roomy columns lost every label it had space for. The cell size comes from the rendered plot area and the label width is *measured* (a 2D canvas at the trace's own font answers exactly how wide `1,284` is) rather than estimated from character counts.

All of it runs from **one `fitChart()`**, in the only order that converges: the ticks decide how tall the axis furniture is, so they settle before the legend measures it. Each step stands down when the author pinned the thing it would touch through `options` (`axisPinned()` beside `legendPinned()`), and each is guarded by `state.fits` so `print` still waits for final geometry.

**Every chart on paper wears a derived `Figure N — <title>` caption** — a bare `Figure N` when the block is untitled, so a chart that renders no caption on screen gains one on paper. The number is never authored: the kernel ships a `figures` map in the payload (`state.figures`), the page binds it by block identity (`state.figByBlock`) and `chartSlotShell`/`renderChartShell` render the prefix into the `.chart-title` slot as text. The **deck always numbers**; the **continuous view numbers only when the canvas declares a `document`** (a report wears numbers, a scratch dashboard does not), so `renderChartShell`/`docHtmlView` take a `numbered` flag while the deck passes `true`. The caption feeds nothing into the TOC — `b.title`, not the prefixed string, is what the TOC-caption fallback receives.

**`recordChartFacts()` is a bystander recorder** run after each chart mounts (and on resize): it reads the DOM the page already produced — the rendered tick labels, the plot-area size Plotly computed, the legend rect — into `state.chartFacts` keyed by the `data-chart` index (`{ticks, elided, axisPx, legendOverlap}`; elided ticks are counted from the ellipsis the runtime appends). It changes nothing — `catTicks` and `fitLegendBelow` are production-fitted and stay untouched — and it exists because the browser is the only party that ever sees rendered geometry. `print` and `snapshot` read `state.chartFacts` at final geometry to report per-figure facts for free (see [cli.md](cli.md)).

Boxes are 320 px by default; `.tall` (460 px) for `scatter3d`/`surface`/`splom`, `.swept` (400 px) when a sweep adds a slider, and both combined (540 px).

## Sweeps

A chart block carrying `sweep` instead of `data` becomes a parameter sweep. `sweepFigures()` builds **one figure per frame** up front, `sweepLayout()` adds a themed Plotly slider (`method: "skip"`, so the step change is a DOM event rather than a Plotly API call — `method: "animate"` is broken upstream for `scatter3d`), and `attachSweep()` listens for `plotly_sliderchange` and swaps the whole figure with `Plotly.react`. The handler lives in `app.js`: **the skill owns the JavaScript, the agent ships only rows.** Because `react` updates in place, dragging a slider across a 3D sweep reuses one WebGL context. `rethemeCharts()` rebuilds every frame on the new palette and holds the reader's current step.

## Document mode: the deck and the packer

A canvas whose envelope carries `document` renders as a **deck of literal page-sized sheets** (`renderDocumentView` in `app.js`). Pagination is by construction: content is packed into fixed-size boxes (`210mm × 297mm` for A4, geometry set through CSSOM), `@media print` resets the screen scale and breaks after each sheet, and a per-canvas `@page { size: …mm …mm; margin: 0 }` rule arrives via a **constructed stylesheet** (CSSOM, exempt from `style-src`). The print engine never chooses a break, so screen and PDF agree — the invariant that carries everything is **`sheet.scrollHeight <= clientHeight`**; a sheet even 3 px too tall silently prints a sliver page.

- **Packer** (`packFragments`): fragments are measured cumulatively inside a hidden auto-height replica sheet at exact content width (the budget comes from a fixed-height probe with the real strips; bodies and fragments are `flow-root` so edge margins cannot escape measurement). Code blocks split by lines via `Range.extractContents` — an hljs span crossing the boundary splits cleanly, DOM surgery, never string surgery; tables split by rows with the `<thead>` repeated; lists by items with `<ol>` numbering continued; paragraphs, charts (fixed-height boxes) and kpi rows are atomic; a heading is never left last on a sheet; an atomic element taller than a page gets its own sheet, clipped (`overflow: hidden` preserves the invariant) with a visible notice. Chapters (`pages[]`) force new sheets.
- **Deck assembly**: the body packs first, then the TOC, so every TOC row can carry the page number its anchor landed on (dotted leaders to the number; digits are written into the already-packed rows — they don't change row height; see canvas-schema.md for the repagination caveat) — then cover + TOC + body + back cover assemble in display order. Header/footer strips are ordinary DOM cloned into every content sheet with `{{pageNumber}}`/`{{totalPages}}` substituted as text after assembly — which is why they survive Cmd+P exactly like the `print` command. On-screen TOC entries scroll to their sheet via `data-doc-anchor` (DOM nodes, never string-built markup).
- **A cover is a sheet, so it can carry a background image.** `applyCoverBackground()` sets `background-image`/`size`/`position` on the **`.sheet` box** through CSSOM — the box, not the padded content box, because a full bleed must reach the paper's edge past the 15 mm margin (`background-clip: border-box`) while the text stays in the padding. CSSOM rather than a style attribute for the usual reason: the CSP drops the latter without a word. The **scrim** is its own absolutely-positioned layer (`.cover-scrim`) between image and text, because a dark photo swallows a near-black title and CSS cannot wash a background in place; **`ink`** repaints the cover's text and *only* the cover's (`theme.text` would paint the whole document, so a white cover title would come with white body text on white paper), and the muted author/date line is derived from it at reduced opacity rather than left to the theme. Z-order: image → scrim → logo/title/subtitle/meta/band. `backCover.background` is the same shape and entirely independent. Neither scrim nor ink is defaulted on — silently tinting somebody's photograph would be presumptuous — so the *validator* warns instead (`COVER_TEXT_MAY_BE_ILLEGIBLE`).
- **Sheets are always light**: `.sheet` redeclares the LIGHT token set, so a dark app renders white paper. Each token the theme can restyle is now wired to its `--doc-*` counterpart with the old literal as the CSS fallback (`--text: var(--doc-text, #1a1d24)`, `background: var(--doc-paper, #fff)`, …), which is what lets an **unthemed** document render byte-identically to before while a themed one repaints. See [Theming](#theming) for the two sinks the theme feeds and what it deliberately does not reach.
- **Paper typography is its own layer, scoped to `.sheet`.** The deck renders each markdown element as its own `.md.doc-frag`, which makes every element both `:first-child` and `:last-child` of an `.md` — so the `.md > :first-child/:last-child` margin reset zeroed the entire document's rhythm, and paper printed with its headings glued to the prose (see [gotchas/frontend.md](gotchas/frontend.md)). The rhythm is re-declared under `.sheet` and loosened for reading on paper (line-height 1.7, generous heading margins), and **code fences wrap** there (`white-space: pre-wrap; overflow-wrap: anywhere`) because a PDF has no scrollbar — an `overflow:auto` fence is not scrollable in print, it is clipped. Wrapping makes fences taller and repaginates the deck; that is safe only because the packer's replica *is* a `.sheet`, so it measures the same wrapped layout the printer produces. **Anything that must apply to markdown on paper belongs under `.sheet`** — that is the one selector both the deck and the measuring replica answer to.
- **White-paper mode is a single-column variant of the deck** (`document.paper`; see [canvas-schema.md](canvas-schema.md#white-paper-mode-a-single-column-academic-variant)). `paper-mode` is a class on the deck **root** (`.doc-mode`), mirroring `data-paper` — so every rule scoped to `.doc-mode.paper-mode .sheet` reaches both the real sheets **and the packer's hidden measuring replica** (both are descendants of that root), which is what keeps the serif/justified/looser-leading layout measured identically to what prints. The **front matter** (title / authors / affiliations / abstract / keywords) is built by `buildFrontMatter()` in `docFragments()` as one atomic fragment that **leads the deck** — ahead of the body, and ahead of a TOC if there were one — so it is the top of sheet 1 without being a standalone page like a cover; the body's first H1 is consumed as the title, not repeated. **Section numbers** (`1` / `1.1`) are prepended into the heading text in `mdFragments()` (`paperSectionNumber()`), so the same numbered text feeds the headings and the TOC from one source; `Abstract` / `References` / `Acknowledgements` stay unnumbered. **Equation numbers** are a separate mount-time DOM pass (`mountEquationNumbers()`) that appends a right-aligned, absolutely-positioned `.eqno` to each `.math-block` in document order — absolute, so it adds no block height and needs no repack; a clean no-op when no math exists. A `## References` list is tagged `.paper-refs` (hanging indent), the footer defaults to a centered page number with **no** header (`docStrips()` seeds it), the default margin widens to ~25mm (`docGeometry()`), and a white paper gets **no auto-TOC** (its front matter is its opening). The continuous view is left minimal — paper is a deck concept and prints as one.
- **A wide table folds instead of losing columns.** The same clip cost a table whole *columns* — an eleven-column table printed with seven and a half — so `tagWideTables()` measures each table in a real sheet body before packing and tags the overflowing ones `.wide`, which switches them to `table-layout: fixed` (the only layout that cannot exceed the measure) with `break-word` cells. Only the offenders: fixed layout divides the page evenly, which would stretch a tidy four-column table across the sheet. Measured with `scrollWidth` — `max-width` clamps the table's box while its columns spill inside it, so `offsetWidth` can never exceed the container and the check would never fire (see [gotchas/frontend.md](gotchas/frontend.md)).
- **No copy button on paper**: `mountCodeCopy(scope, {button: false})` mounts the `.code-block` wrapper without its button, so the deck holds none at all (nothing to hide at print time) and the fence reclaims the right padding the button reserved. The wrapper still goes on *before* the packer runs — a fence's geometry must be settled before it is measured, never after. The continuous view keeps one button per fence.
- **Deck ⇄ continuous toggle** (topbar segmented control, visible for **every** loaded canvas — the view is presentation, not capability). A declared `document` opens as the deck and everything else opens continuous — but that is the *canvas's* default, and **the reader's choice outranks it and is sticky across navigation**. `state.docViewChoice` (null = follow each canvas's own default) records the view the reader clicked, and every canvas opened afterwards honours it: browsing a folder of markdown as paper is one click, not one click per file. A canvas that cannot be decked at all — a form, a confirm, a sweep — falls back to continuous **for that canvas without forgetting the choice**, so the next deckable document comes up as paper again. A display canvas builds its deck **lazily on first toggle** (`renderCanvas` branches on `state.docView`; packing a 20-chart gallery eagerly on every open would be waste). Every deck derives its defaults — the TOC is auto-generated from the document's **headings and chapter names** whenever anything exists to list (`wantToc`; the JSON `toc` key only customizes title/depth). **Block titles are captions, not structure, and are not listed** — interleaving them with the headings printed a numbered outline broken up by unnumbered caption rows. They remain the *fallback* (`tocEntries()`): a canvas with no prose at all has no other structure, so a chart gallery keeps its contents page, A4/15mm, LIGHT palette via `state.docLand` in `palette()`. A topbar contents button (`#tocBtn`) lets the reader toggle the TOC off and on — a repack, since the TOC's own sheets shift every page number after them. A second button (`#stripsBtn`) does the same for the **running header/footer**, which is derived rather than declared: `docStrips()` resolves a three-state reader override (`state.docStrips`: null = auto, on iff the canvas declared strips) into the strips actually rendered, deriving the canvas title and `{{pageNumber}} / {{totalPages}}` when nothing is declared, and letting the reader drop an author's pair just as readily. It repacks for a sharper reason than the TOC does: `packFragments` probes its per-sheet budget from a sheet **with the strips attached**, so they consume content height — turning them on can add a sheet, which moves every page number after it. Both toggles are per-canvas (`renderCanvas` resets them on navigation — deliberately unlike the view choice above, which follows the reader: a TOC and a running header are opinions about *this* document, while deck-or-continuous is an opinion about how one reads) and live in memory only, so `instant-canvas print` — a fresh page load — renders the declared JSON and never sees them; Cmd+P from the reader's own browser does. Interactive canvases keep the toggle with the deck side muted (`.vt-off`, still clickable): the click shows a toast naming the blockers (`deckBlockers`) and why paper refuses them. A **floating print button** bottom-right (`.print-fab`) appears whenever a deck exists (`window.print()` → the native dialog where "Save as PDF" lives; fires `beforeprint`; hidden in print output). Once in document land, both views live in the DOM with one hidden by a class. **Charts exist once** — each view renders an empty `.chart-slot`, and the live plot node moves between slots (`moveChartsTo` + `Plotly.Plots.resize`, never purge + newPlot). Print CSS always shows the deck and hides the continuous view, so Cmd+P prints sheets no matter what is on screen; `beforeprint` relocates the chart nodes into the deck (a `.printing` class keeps the hidden deck laid out off-screen so resize sees real sizes) and `afterprint` restores them. If `beforeprint` ever failed to fire, pagination would still be correct — only chart sizing could degrade.
- **The paper-only controls disable, they do not vanish.** `#tocBtn` and `#stripsBtn` are properties of paper: off the deck there is no sheet to put a header on and no page numbers for a TOC to cite. They used to be `hidden` in the continuous view, which is wrong twice over — the toolbar reflowed as they came and went, moving every other control out from under a cursor already reaching for one, and a reader who never saw them had no way to learn they existed. `paperControl()` in `syncViewToggle()` keeps them in place, sets `disabled`, strips the `active` ring (an "on" ring on a control you cannot press is a lie about the current view) and writes a `title` naming the view they need — with two distinct reasons for the TOC, which is also unpressable *on* paper when the document has no headings to list. This is the rule `#viewDeck` beside them already followed: **a hidden control teaches nothing.** `#paletteBtn` is deliberately **not** in this group (see below). **`#paperBtn` — the white-paper convert toggle — joins this group** (`document.paper`, [architecture.md](architecture.md#where-a-saved-theme-lands--libthemestorejs-the-one-write-path) for its write path): it is a persistent write, not a per-tab view, so like the theme it must reach `print` — but it is a paper property, so it disables off the deck with the same reason. On the deck it is **lit when the document is a paper** (click to revert to a normal document) and unlit otherwise (click to convert); it **hides** entirely only where paper is impossible either way — a form/confirm/sweep, an image/media/presentation, or a canvas that already declares a cover.
- **Colors are not a paper-only control, and the continuous view wears the brand.** A theme belongs to the *document*, not to the deck, so `#paletteBtn` stays live wherever a canvas is loaded. That only means anything because the continuous view actually takes the theme: `applyDocumentTheme()` is applied to whichever root is on screen — the deck's `.doc-mode` or the classic render's plain `.canvas`, both of which match `.canvas` — and `palette()` hands charts the document's colorway over the *app* theme's ink. What the continuous view declines is the **paper**: `paper`/`text`/`surface` are sheet tokens, and forcing them on a dark app paints black on black, so only `--accent` and `--doc-link` cross over (`styles.css`, `.doc-mode .doc-html, .canvas:not(.doc-mode)`). The colorway override is gated on a theme having actually been **declared** (`themeIsDeclared()`), because the default colorway is the light one and forcing it on a dark app would hand it back a palette it deliberately does not use. Without any of this, opening the picker off the deck would have given the reader a color picker that visibly does nothing.
- **Save says what it will do, before it does it.** A colour click can now make a *file appear* in the reader's repository — the companion canvas that gives a markdown document an envelope. `GET /api/theme/plan` (fetched when the panel opens, and re-fetched after every save) answers *what would this write touch*, and `#palNote` renders it: **"Save will create README.canvas.json"**, or a warning that a display canvas will gain a `document` and start opening as paper. On a canvas holding a form, a confirm or a sweep, the plan comes back `blocked` and **Save goes disabled with the reason in its `title`** — visible, not hidden, because a control the reader cannot see teaches nothing — while "All documents" beside it still works. That is the honest way out: an interactive canvas cannot carry a `document`, so the workspace default is the only theme it can wear.
- **The palette control** (`#paletteBtn` → `#palettePanel`) is **two screens**, because it does two jobs and conflating them made both illegible. **Pick** (`#palPick`, the default): the preset chips with swatch strips, grouped **Light paper** / **Dark paper** (`#palPresets` / `#palPresetsDark` — a reader who wants a dark deck has already decided, and should not have to read fourteen names they do not want), then the workspace's own palettes, then the document footer — Reset / "All documents" / Save. **Edit** (`#palEdit`): every token, the chart colorway, a name, one Save. A round **+** in the header (`#palAdd`) is the only way in and the whole answer to *"how do I add my own colors?"*; `#palBack` is the way out; `state.palView` says which screen is up and `showPaletteView()` swaps the body. Three rules hold it together, each learned by getting it wrong first: **(1)** the "+" is unconditional, so an empty workspace still advertises the feature — hiding the custom section when there were none meant deleting your last palette deleted every trace that palettes existed. **(2) One screen, one Save.** The document's footer is *hidden* in the editor; leaving it up sounded principled (*the footer always means the document*) and rendered as two identical primary buttons both reading "Save", one writing the document and one the workspace, twelve pixels apart. **(3)** The editor starts from the colors on screen and, if those are one of the reader's own palettes, from its **name** — so saving *updates* it (the button reads `Update`) rather than forking a second near-identical "My brand", and "edit a palette" needs no separate affordance. Saving returns to the list where the new chip sits lit: the chip is the confirmation. **`#palDetail`** sits below the list and outside the scroller — what the selection *is*, so reading it never costs a scroll: name, every token, every chart color, and for a preset the guidance (`description`) that otherwise lives only in a chip's `title` tooltip, where the one thing that would make a reader choose Okabe-Ito over Ember (*colorblind-safe by construction*) was unreadable. `paletteIdentity()` distinguishes three states, and the third is the one the old panel could not express: a preset, one of the reader's own palettes, or **a preset with overrides on top — which is no longer that preset**, and says so instead of leaving its chip lit and staying quiet. Every edit previews live — `previewTheme()` repaints the `--doc-*` properties and calls `rethemeCharts()`, in place, for the same WebGL-context reason the app theme toggle does — and **Save** is what persists it, through `POST /api/theme`. Picking a preset chip *drops* the token overrides, because a chip that does not give you what it shows is a lie. Reset on a canvas that declares its own `document.theme` refuses with a toast rather than editing the author's file (the kernel says the same thing with a 409). Nothing in the page knows what a preset *is*: the kernel hands it hex (see [architecture.md](architecture.md)), and `resolveLocally()` mirrors the layering rules only far enough to preview an unsaved edit before the round trip — but it must mirror them *exactly*, because a preview that disagrees with the kernel is a lie about what Save is going to do. The panel decides whether a click was inside itself in the **capture phase**, which is not an optimization but the only thing that works (see [gotchas/frontend.md](gotchas/frontend.md)).
- **The colorway editor has two subtleties, both consequences of the one-is-a-lead rule** (see [canvas-schema.md](canvas-schema.md)). A swatch can be removed down to a **floor of two**: one color is a *lead* the preset would refill from, so removing the second-to-last would silently undo itself. And the colorway a reader edits is first **materialized** (`currentWay()`) — until they touch it the declared theme may name no palette at all, the preset having supplied one, and an edit to a colorway that is not there cannot be expressed. The first touch pins the whole resolved colorway; every edit after it is exact.
- **Custom palettes are a library, not preset names.** Saving one captures the **resolved** theme, never the declared one (`savePalette()`): a library entry that said only `{preset: "forest"}` would be a pointer rather than a palette, and would repaint under the reader if the preset ever changed. Applying one materializes its colors into the document, so nothing is left in the file to match a chip against by name — `activeCustomName()` therefore matches **by value**, deep-comparing the declared theme against each entry, and a matched custom chip lights up while *no* preset chip does. Deleting an entry (`POST /api/theme/palette` with `theme: null`) repaints nothing: the colors were copied into the documents that use them, and they are still what those documents say. [canvas-schema.md](canvas-schema.md#custom-palettes-a-library-not-a-preset-name) carries why the canvas must not hold the reference. **The reader is not the only author of this library**, which is why `state.themePresets` — fetched once per session, the workspace's palettes riding in it — is **dropped on every `workspace` broadcast**: an agent running `instantcanvas theme --save` writes a palette straight to disk, and a picker still showing the cached list would be showing a library that no longer matches the file it came from.
- **This is deliberately not a third reader toggle.** The TOC and header/footer buttons live in memory and die with the tab, because they are opinions about how to *look* at a document. A theme is part of the document: it must survive a reload, reach `print` (a fresh page load, which never sees a reader toggle), and be readable by the agent that wrote the canvas. Hence a persisted setting with an explicit Save, not a per-tab switch.
- Screen presentation: sheets on the app background with shadows, scaled to fit via one `transform: scale()` on `.deck-scale` (CSSOM; print resets it with `!important`), refit on resize through a ResizeObserver.

## Presentation mode: the filmstrip and the stage

A canvas carrying a `slides` array is a **presentation** — a *sibling* of the document deck, not a variant of it. `renderCanvas` branches on `Array.isArray(canvas.slides)` and hands off to `renderPresentationView` **before any of the deck/continuous machinery** runs, because a deck of slides shares none of it: no packer, no continuous twin, no TOC. `state.presLand` is the flag, mirroring `state.docLand` exactly — and `palette()` hands charts the document colorway over its own surface whenever `docLand || presLand`, so a slide charts like a sheet.

- **The filmstrip is the browse view.** Each slide is a fixed-geometry box — `1280×720px` for 16:9 (`13.333in × 7.5in`) or `960×720px` for 4:3 (`SLIDE_GEO`/`slideGeometry`) — stacked vertically and scaled to fit the pane by **one** `transform: scale()` set through CSSOM (`fitStrip`, the deck's `.deck-scale` pattern; a ResizeObserver refits). Sizing is in **inches**, the same unit as the `@page`, so print is 1:1 with no sub-pixel doubling. The browse chrome sits **below** each box, *unscaled* so it stays readable at any fit: a muted `Slide N of M` label, an overflow badge, and the slide's `notes` paragraph.
- **Seven class-based layout renderers** (`SLIDE_HTML`: `title`, `section`, `content`, `two-column`, `quadrant`, `statement`, `closing`) build each layout's grid, and its regions fill with the very same block renderers the rest of the app uses — `renderMarkdown`/`renderKpi`/`renderTable`, plus a chart-box for charts. A lone chart or KPI **fills** its region (the CSS flexes it; the ResizeObserver `mountCharts` attaches drives `Plotly.Plots.resize`). Backgrounds reuse `applyCoverBackground` on the slide box (image → scrim → content — z-index on a flex item needs no `position`, the `.sheet` cover lesson). The declared running footer is cloned into every slide but title/closing via `slideFooterHtml`, with `{{slideNumber}}`/`{{totalSlides}}` substituted as **text**.
- **A slide is its own surface, like a sheet.** It takes the **full** `--doc-*` token set — paper included — via `applyDocumentTheme` on the `.pres-mode` root, so a dark deck is normal and `data-paper="dark"` swaps the semantic layer. The `.slide` CSS layer in `styles.css` is scoped exactly like `.sheet`, but with its own **large** type scale: a slide is read from meters away, not from a chair.
- **Autofit (D6) measures after the DOM settles**, the same discipline as the packer. `autofitSlides` awaits `document.fonts.ready` first — a badge that measures a half-laid-out slide lies — then `autofitOne(slide)` checks each `.slide-region` for `scrollHeight > clientHeight` and steps the type scale down through **at most three** class-based steps (`.fit-1/2/3` → `--slide-fs`). Still overflowing after that, it adds `.clipped` and reveals a **filmstrip-only** badge — the reader sees the overflow while browsing, and the stage never shows a warning to an audience.
- **The topbar reflects that a deck has no continuous twin (D9).** A `#presentBtn` (**Present**) owns the view-toggle slot in place of the deck⇄continuous pair; `#tocBtn` and `#stripsBtn` **disable with reasons** (a presentation has no table of contents and no running-strip toggle — its footer is declared, not toggled); `#paletteBtn` stays **live** because colors belong to the document; and the print FAB shows. `syncViewToggle` carries a dedicated `presLand` branch for all of this. The sidebar badges a deck with a distinct Lucide `presentation` glyph (`scan.js` sets a `deck` flag; the `kind` stays `canvas`).
- **The presenting stage (Phase D) is a sibling `#stage` overlay** — one slide filling the screen, entered fullscreen via `requestFullscreen()` on the stage, or **in-viewport** when the browser refuses the gesture (D10, never a hard dependency), scaled to contain the viewport by one transform (`fitStage`). `presentStart` opens on the **most-visible** filmstrip slide. The live chart nodes **move** from the filmstrip into the stage slide on entry and back on exit (`stageMoveChartsIn`/`stageReturnCharts` + `Plotly.Plots.resize` — never purge + `newPlot`, so no WebGL context leaks), while each stage slide is built **fresh** (`buildSlide`) and autofit-measured (`autofitOne`). Keyboard control (D11) is scoped to presenting by a **capture-phase** handler that swallows *every* key — so `⌘K` and `/` keep their meaning only elsewhere and do nothing here: →/↓/Space/PageDown/Enter and a click advance; ←/↑/PageUp/Backspace go back; Home/End jump to the ends; typed digits + Enter jump by number; `B` toggles a black overlay; `Esc` exits. The cursor auto-hides after ~2s idle (`.cursor-hidden`) and wakes on move. A native fullscreen exit returns to the filmstrip **at the current slide**; navigating to another canvas leaves presenting; and a hot reload holds the slide index (clamped to the new count) and re-grabs the fresh chart nodes the rebuilt filmstrip just mounted.

## Bespoke form widgets

Everything the user touches is custom — native browser chrome never appears mid-form:

- **date / datetime picker**: calendar popover with month/year quick-select grids (12-year paging), Today/Now/Clear; datetime adds an hh:mm section and a Done button (day-pick keeps it open). The input stays typable ISO, so native `required`/`pattern` validation still applies.
- **select**: styled menu popover, check on the current option, Lucide chevron aligned with the other input icons; the display input is a keyboard-openable trigger (Enter/Space/↓) whose typing is suppressed.
- **radio / checkbox / range**: `appearance: none` rebuilds — accent dot, SVG-check box (data-URI, allowed by `img-src data:`), slider with accent progress fill (`--fill` custom property set from JS).
- **`ui: "buttons"`** segmented control and **`ui: "pills"`** token multi-select (filter input, removable ×) — both store values in hidden inputs so `collectValues()` reads them uniformly.
- **Validation UX**: live on-blur checks (`clientFieldError()` mirrors the kernel's rules — email, URL + protocol whitelist, pattern with `patternMessage`, ranges, dates) render into inline error slots and clear on input. The kernel re-validates on submit regardless; 422 responses render field-level errors, 409 responses drive the overwrite/outside-root confirmation dialogs.

- **The native `.env` form** (`renderEnvForm`, keyed off the envelope's `envNative` flag; see [canvas-schema.md](canvas-schema.md) and [security.md](security.md)) is its own form renderer — one row per key: a fixed monospace KEY (an editable name-input for an added row, `pattern` = `ENV_KEY_RE`), the value, a **copy** toggle and a **delete** control. Several deliberate choices:
  - **Values are masked by default** (`type="password"`), the eye reveals per row. The system cannot know which values are secret, so the safe default hides every one — the field stays `type: secret` server-side; only the widget's default display flipped.
  - **Self-service, session or not.** No buttons are disabled: with an agent session (`open .env`) Save resolves it; without one (the reader clicked the file in the browser) Save posts to `POST /api/env/save` — a direct disk write (see [architecture.md](architecture.md)). Cancel with no session just navigates back to the folder.
  - **Add / delete are DOM-surgical, never a re-render** (the re-render-detaches-the-clicked-node family, [gotchas/frontend.md](gotchas/frontend.md)): "add" appends a row, "delete" toggles an `env-deleting` class (struck-through, the trash swapped for an undo ×) so a deletion stays **visible and reversible** until submit — a count/list in the delete confirmation is a promise. `collectEnvSubmit` reads `{values, deletions}` off the DOM at submit time.
  - **Copy-to-clipboard is an additive toggle.** Each copy button flips its row's membership; `syncEnvClipboard` rewrites the whole clipboard (`navigator.clipboard.writeText`, secure-context on `127.0.0.1`) from every toggled row's current `KEY=value` (quoted like the writer), so three clicks put three pairs on it and un-clicking removes one. A **paste** of `KEY=value` text adds/updates rows (`parseEnvPairs` + `applyEnvPaste`), while a single-line paste into a value field still just fills it.
  - The save-success modal drops the raw redacted-result JSON for the env form (`showSuccess(payload, {minimal})`) — noise for a human editing their own file; agent forms keep it. A muted one-line hint under the security banner teaches the copy-then-paste flow.

## Layout

`.canvas` has **no max-width**. It is a fixed gutter (`--gutter`, 40px, narrowing to 16px under 720px) and everything else is content — markdown, tables and charts all fill the pane. Tune `--gutter`; do not reintroduce a measure cap.

Fieldsets render as bordered groups (`--fset-border` token) with 1–3-column grids; below 600 px those multi-column grids collapse to one column. **CSP blocks `style=""` attributes**, so all geometry is class-based: `.cols-2/.cols-3` on the grid, `.span-2/.span-3` on fields, utility classes elsewhere. JS may set `el.style.*` (CSSOM is allowed); markup may not carry style attributes.

Every code block rendered **on screen** gets a copy-to-clipboard button. `mountCodeCopy()` runs after mount and wraps each `<pre>` in a `.code-block` positioning context, because the button is chrome rather than document content and must not travel with the markdown. It is **always visible, never hover-gated** (see [gotchas/frontend.md](gotchas/frontend.md)). On paper it is not mounted at all — `{button: false}` puts the wrapper on and nothing else, since nobody copies a PDF to the clipboard. `copyText()` prefers `navigator.clipboard` — `127.0.0.1` is a secure context — and falls back to an off-screen `<textarea>` plus `execCommand`, positioned by the `.offscreen` class rather than a style attribute.

## Theming

**Two different things are called a theme, and they never mix.** The **app theme** is the reader's light/dark preference for the chrome around the content — a per-tab UI opinion. The **document theme** is the brand a *document* carries onto paper: it is resolved by the kernel, persisted to disk, and printed. A sheet is paper in a dark app just as it is in a light one.

The app theme is two mechanisms, deliberately separate:

- CSS custom properties per theme (`:root`, `[data-theme="dark"]`, and the `prefers-color-scheme` fallback), including input tokens (`--inp-*`), fieldset border, syntax-highlighting tokens (`--code-*`, which carry the hljs theme — light mode needs darker hues than the chart palette to stay readable on `--panel-2`), and `color-scheme: light/dark` so native widget internals (number spinners, scrollbars) follow the theme. **All three theme blocks must be kept in step.**
- **Plotly cannot read CSS variables**, so two concrete palettes (`LIGHT`/`DARK`) compile into a `layout.template` via `plotlyTemplate()`. The default colorway is HappySkills orange-led — `#eb4a26 #2e6fd8 #0e9384 #9b51e0 #d6336c`, validated (dataviz six-checks) for colorblind separation and surface contrast on both white and dark paper, so both modes share the one set. The toggle calls `rethemeCharts()`, which rebuilds each figure and applies it with **`Plotly.react` — in place, never `purge` + `newPlot`**: WebGL contexts are never released on teardown, so rebuilding would exhaust the browser's context ceiling (measured: 6 toggles → 6 contexts leaked vs 1 reused). Nothing else needs re-rendering; the CSS variables carry it.

The document theme arrives from the kernel already resolved — every key a literal hex string, no preset names, no layering rules in the browser (`lib/theme.js` owns them; see [canvas-schema.md](canvas-schema.md)). It reaches the page through **exactly the same two sinks, for exactly the same reason**:

- **CSS**, as `--doc-*` custom properties written by `applyDocumentTheme()` onto the `.doc-mode` root through **CSSOM `setProperty`** — the CSP drops `style=""` attributes but exempts programmatic assignment. `.sheet` reads them for its own token set, so everything the sheet already styled with `--text`/`--accent`/`--border` follows for free. That CSSOM route is also precisely *why* strict-hex validation matters: `setProperty` will happily take the string `javascript:alert(1)`, so the value is checked by the validator and re-checked at the `/api/theme` boundary, because the browser is not trusted (colors are only ever handed to `setProperty`, never interpolated into markup).
- **Plotly**, through `documentPalette()`, which compiles the resolved colorway into the figure template — the same "Plotly cannot read `var()`" constraint as above, which is the entire reason a colorway must be concrete hex by the time a chart is drawn rather than a CSS variable like everything else. `palette()` branches on `state.docLand`: in document land a chart paints the document's theme over a light sheet whatever the app theme says.

Two boundaries are load-bearing. **Semantic colors are not themeable** — green/amber/red (a KPI delta means something) and the `--code-*` syntax palette stay put; they carry meaning and are not the author's to rebrand. And **the continuous view takes the brand but not the paper**: `.doc-mode .doc-html` maps `--accent` and the link color and stops there, because it is a screen view that still follows the app's light/dark theme — forcing a document's `paper`/`text` tokens onto it would print black on black.

## CSP shim

`csp-shim.js` loads before `plotly.min.js` and reconciles Plotly with `style-src 'self'`: it plants a `.no-inline-styles` stub so Plotly skips its runtime `<style>` injection (the rules come from the vendored `plotly.css` instead), and reroutes `setAttribute('style', …)` — which the colorbar uses — into CSSOM assignment, which the CSP exempts. Verified in a browser under the real kernel headers: zero violations, zero injected `<style>` elements.
