# Changelog

## [Unreleased]

### Added
- **The companion canvas: a markdown file can finally carry a cover, a theme, a running header — anything a document carries.** A `.md` has **no envelope**. It *is* the canvas: the runtime synthesises one in memory and never writes it, so there is nowhere to keep any of that. Everything a document might want beyond its prose lives in `document`, and a markdown file cannot hold one.

  The first answer was a bespoke dotfile holding a per-path theme map. It worked, and it was wrong in the way that matters: **it only ever solved colour.** A cover could not go in it. Neither could a back cover, a running header, or page geometry. Every new furnishing would have needed a new bespoke key — reinventing, badly, the canvas envelope that already existed.

  The insight that resolves it: **the thing a `.md` is missing is a canvas. So give it one.**

  ```jsonc
  // README.canvas.json — beside README.md
  {"instantcanvas": 1, "enhances": "README.md",          // ← one new envelope key
   "document": {"cover": {…}, "theme": {…}, "footer": {…}},
   "blocks": [{"type": "markdown", "src": "README.md"}]}
  ```

  It is an **ordinary canvas** — nothing new to validate, nothing new to learn, and every `document` furnishing worked the day it shipped, because it already did. One key buys the entire envelope.
  - **The companion is what runs, everywhere and uniformly.** `open README.md` renders it, `print README.md` prints it, and the sidebar shows **one** entry — the document, badged. All three arrive through the same `loadCanvas`, which is what makes it free. Anything else is a trap: a reader who sees a cover on screen and no cover in the PDF has been lied to.
  - **Declared, never sniffed.** The companion is found by reading `enhances`, not by scanning blocks for a markdown `src` that happens to match. Sniffing is ambiguous and would bite — a genuine report that quotes the README among its other content would hijack the README's entry, and nothing could tell *"this is README's metadata"* from *"this is a document that happens to include README"*. A declared key cannot be ambiguous and **survives any rename**: `<base>.canvas.json` is only what the runtime writes by default, for humans.
  - **Two canvases may not enhance one file** (`DUPLICATE_ENHANCES`, naming both). First-wins is a coin toss the reader cannot see.
  - **Saving a theme on a bare `.md` creates the companion** — a visible, tracked file in the user's repo, which is honest, portable, and reviewable in a pull request. It is also a file appearing from a colour click, so **the UI says so before it happens** (a new `GET /api/theme/plan`: *"Save will create README.canvas.json"*), and so does the CLI.
  - **A companion supplies an envelope, not an author.** Its `markdown` block points at the user's own README — badges and all — which is the *authored* `src` path, where a remote image is a hard `REMOTE_ASSET_BLOCKED`. Branding a README with a shields.io badge in it would therefore have produced an invalid canvas and **stopped the document rendering at all**: the reader picks a colour and breaks their own README. A companion rendering *its own document* now degrades exactly as `open README.md` degrades. With or without a companion, the same file renders the same prose; only the furnishings differ.
- **A cover is a sheet, so it can carry a background image.** `cover.logo` is a 48 × 48 **mark** — a photograph put through it renders as a postage stamp, which is not a cover image by any definition. `cover.background` is the real thing: a full-bleed photo, edge to edge, reaching past the 15 mm margin to the paper's edge. `backCover.background` is the same shape and **entirely independent**.
  ```jsonc
  "background": {"src": "assets/hero.jpg", "size": "cover", "position": "25% 50%",
                 "scrim": {"color": "#000000", "opacity": 0.35}, "ink": "#ffffff"}
  ```
  - **One concept, both use cases.** `size` + `position` is the CSS background model, which already expresses *"fill the sheet"* and *"place a sized image somewhere"* — so there is no second mechanism. Percentage `position` is a **focal point**, not an offset: it decides which part survives the crop. The subtlety worth knowing, because it catches people: **it only moves the axis the image actually overflows.** On portrait A4 a square photo *and* a landscape photo both overflow sideways, so the horizontal number is almost always the live one.
  - **Legibility is not optional, and it does not solve itself.** A dark photo swallows a near-black title, and `theme.text` cannot fix it — that token paints the *whole* document, so a white cover title would come with white body text on white paper. Hence two cover-scoped knobs: a **`scrim`** (`{color, opacity}`, so the strict-hex rule everything else obeys still holds) and an **`ink`**, which repaints the cover's text and nothing else, muted author line included. Neither is defaulted on — silently tinting somebody's photograph would be presumptuous — so the validator **warns** instead (`COVER_TEXT_MAY_BE_ILLEGIBLE`).
  - It reaches the **PDF**, not just the screen, and an oversize image is a hard `ASSET_TOO_LARGE` rather than a silent truncation: a full-bleed photo is paid for twice, in the canvas payload and in the PDF, and nobody should ship a 40 MB PDF by accident.
- **`theme` — the agent's door to the color system, which it did not have.** The use case is the obvious one and it did not work: a user asks their agent to style a report in the company's brand colors, the agent reverse-engineers them from the website, and then has to *set* them. For a canvas the agent authored that was always possible — write `document.theme`, and `validate` type-checks every color. A **native `.md` had no envelope to write into at all**. Now it does — its *companion canvas* — and this command is how an agent creates one without having to know the shape.
  ```
  theme <canvas.json|file.md>              # what is it wearing, and WHICH FILE decides?
  theme <file> --set '{…}' [--clear]       # a .md gets a COMPANION, created and named first
  theme --all --set '{…}'                  # the workspace default (no file argument)
  theme --save "Acme" --set '{…}'          # a reusable palette; appears in the reader's picker
  theme --list                             # every preset and every saved palette
  validate skills-config.json              # the colors in the project's own config
  ```
  - **Where a theme lands is decided, not asked** — and by the same rules the browser uses, because they are now literally the same code (`lib/themestore.js`). Two doors, one implementation: a reader clicking Save and an agent running `theme` cannot disagree about where a theme belongs. It always lands in the document's **own envelope**: a canvas's `document.theme`, spliced in as text; a markdown file's **companion canvas**, created if absent; a display canvas's `document` object, created for it. A canvas holding a form, a confirm or a sweep is **refused** (`THEME_NEEDS_DOCUMENT`) — `document` is invalid beside an interactive block, and a colour click must never make the agent's own canvas stop validating. The form is the form; it wears the workspace default.
  - **A color that is not strict hex is refused, never dropped.** `resolve()` is deliberately forgiving — it has to be, since it also runs on a hand-edited config — and would have quietly discarded a brand color scraped as `crimson` or `rgb(228,0,43)`, leaving the agent to report success on a theme that did not take. The write boundary refuses instead, names the offending path, and lists the known presets back on a typo.
  - **A theme written by the CLI repaints an open browser.** Most writes ride `fs.watch` — a companion is an ordinary `*.canvas.json` — but `skills-config.json` may sit *above* the workspace root, where the watcher cannot see it at all. `POST /api/refresh` closes that, best-effort: no kernel running is the normal case, not an error.
  - **`validate` also checks the colors in `skills-config.json`**: a bad preset name, a CSS color, a palette that shadows a built-in. (It does not quote the file's bytes back — the same rule that stops `validate .env` printing a secret into the agent's context — and it says to fix the file *in place, never by deleting it*, because it holds every skill's settings.)
- **Document colors are now a system, and the reader owns them too.** `document.theme` accepted an `accent` and a chart `palette` and nothing else, which left the two most useful cases unreachable: an agent had to invent a coherent five-color scheme from scratch to brand anything, and a **native markdown file could not be branded at all** — a `.md` *is* the canvas, its envelope is synthesised in memory and never written, so there was no `document` object anywhere to hold a theme. Both are fixed by the same contract. A theme now starts from a named **preset**, each supplying an accent *and* a matching chart colorway, and any of seven tokens — `accent` `paper` `surface` `text` `muted` `border` `link` — overrides it on top. `$IC catalog theme`.
  - **Twenty-two presets, in two groups.** On **light paper**: `default` `slate` `ocean` `forest` `plum` `ember` `mono` `sepia`, plus the ones an audience already reads — `tableau` (Tableau 10), `okabe` (Okabe-Ito), `carbon` (IBM Carbon), `nord`, `solarized`, `material`. A reader asking for "the Tableau colors" is not asking to be designed for; they are asking for the palette their audience has been reading for a decade. On **dark paper**: `midnight` `graphite` `abyss` `moss` `dracula` `tokyo` `solarized-dark` `okabe-dark`. The picker groups them, because a reader who wants a dark deck has already decided and should not have to read fourteen names they do not want. Choosing on grounds other than taste: **`okabe`, `okabe-dark` and `carbon` are colorblind-safe by construction**, and `mono` is the only preset that survives a black-and-white printer.
  - **"Dark" is not a flag — it is a paper color, and everything else is derived from it.** No key declares a theme dark. The sheet's entire dark set — its code syntax palette, its card surfaces, its accent wash, and the Plotly template its charts compose over — follows from the *luminance* of the resolved `paper` (Rec. 709, so `#0000ff` is dark and `#ffff00` is not). Which means `{"preset": "forest", "paper": "#101010"}` is a dark document, a custom palette with dark paper is one too, and a reader who simply drags the paper swatch to near-black in the editor gets all of it — with no second field restating what the first one already said, and no way for the two to contradict each other. The alternative was a `mode` key that could disagree with the color sitting next to it.
  - Tokens alone were never going to be enough, which is the whole reason this is more than a list of hexes: a `#6d28d9` keyword on a `#282a36` sheet is *invisible*. The semantic layer is not the author's to rebrand — green means up, red means down — but it does have to be legible, so dark paper swaps the whole set wholesale. And **dark paper prints dark**: the deck is the printed page and `print` renders backgrounds, so a dark preset is a full-bleed dark PDF. Right for a screen, expensive on paper. The catalog says so, and so does the panel, rather than letting the reader find out at the printer.
  - **The theme colors the charts too**, so a document and its plots can no longer disagree about what the brand color is. Two rules make that hold without a schema for it. An `accent` declared with no `palette` of its own **leads** the colorway (the old runtime did this, and losing it gave you a blue heading over a green first series). And within `palette`, **one color is likewise a lead** the preset fills out — so pinning your brand color does not paint all five series the same blue — while **two or more *are* the colorway, exactly as given**. (The first cut of this extended *any* short palette from the preset. That made a deliberate three-color chart inexpressible, and would have made removing a swatch in the browser's colorway editor silently refill itself.)
  - **A workspace can carry its own palettes.** `skills-config.json` holds a `"palettes"` map under our `owner/name` key, offered in the browser beside the built-ins and writable by an agent for a user with brand colors. It is a **library, not a set of new preset names**: applying one *copies its colors into* `document.theme` rather than leaving a `"preset": "My brand"` reference behind. That is the load-bearing decision. A canvas is a self-contained contract — an agent reading it must see the actual colors, `validate` must stay a pure function of the file, and **a canvas mailed to someone else must not silently repaint itself against *their* workspace config**. The cost is paid in the picker, which therefore has to match the active chip by *value*; the benefit is that a canvas never carries a reference it cannot resolve on its own.
  - **Precedence is three levels**: the document's own `document.theme` → the workspace default (`theme` in `skills-config.json`) → the built-in default. The document always has the last word, because a theme an agent wrote *into* a canvas is part of that canvas's contract and a workspace default must not silently repaint it.
  - **A palette control in the browser, which writes its choice back to the file.** This is deliberately *not* a third reader toggle. The TOC and the running header/footer live in memory and die with the tab, because they are opinions about how to *look* at a document. A theme is part of the document: it has to survive a reload, reach `print` (a headless Chrome that never sees a reader toggle), and be readable by the agent that wrote the canvas. So it persists — preview freely, Save on purpose — which is also the only reason `print` picks it up for free.
  - **The reader cannot delete a canvas-declared theme from the browser** (`THEME_DECLARED_IN_CANVAS`, 409): the canvas is the author's contract, and the runtime does not edit it out from under them.
  - The theme is resolved **server-side**, to literal hex, before the page ever sees it (`lib/theme.js`). The browser never learns what a preset is, so there is no second copy of the color rules to drift from the first — and `print`, being the same page in a headless Chrome, inherits the answer for nothing.
- **The paper-only controls go disabled off the deck instead of vanishing.** The TOC toggle and the running header/footer toggle are properties of *paper*: in the continuous view there is no sheet to put a header on and no page numbers for a TOC to cite. They used to be `hidden`, which is the wrong answer twice over — the toolbar reflowed as they came and went, moving every other control out from under a cursor already reaching for one, and a reader who never saw them in the continuous view had no way to learn they existed at all. They now stay in place, dim, unpressable, and carry a `title` naming the view they need. This is the rule the deck button beside them already followed, and its own comment already said: *a hidden control teaches nothing*.
- **The continuous view wears the document's brand too, so the palette control is live in both views.** Colors are emphatically *not* a paper-only control, and the first cut of the rule above wrongly treated them as one. A theme belongs to the **document**, not to the deck: the continuous view now takes the same accent, links and chart colorway, and declines only the *paper* — which a dark app would render as black on black. Previously the theme had no root to land on at all in the classic render (there is no `.doc-mode` element there), so a reader who opened the picker off the deck would have been given a color picker that visibly does nothing. The tokens are applied to whichever root is on screen, and `palette()` hands charts the document's colorway over the app theme's ink — but only when a theme was actually *declared*, since the default colorway is the light one and forcing it on a dark app would hand it back a palette it deliberately does not use.
- **The document/continuous toggle now follows the reader, instead of resetting on every file.** The view was decided per canvas — deck if it declared `document`, continuous otherwise — and re-decided on every navigation, so "read this folder as paper" meant clicking the toggle once per file, and clicking it was the one thing that could not make it stick. The reader's choice now outranks each canvas's default and survives navigation. A canvas that *cannot* deck (a form, a confirm, a sweep) still falls back to continuous for itself without forgetting the choice, so the next deckable document is paper again.
- **The workspace path in the topbar copies itself.** It is the one string a reader routinely wants *out* of this app and into a terminal or a file dialog, and it was displayed in a `<span>` that could not be usefully selected — it is elided from the head (`…/scratchpad/ws`) to fit the bar, so even a careful drag-select yielded a truncated non-path. It is now a group: the path is a click target, and an explicit copy button sits on the end to advertise that it is one, since an affordance nobody can see is not an affordance. **What lands on the clipboard is always the full path, never the text on screen** — which is the whole point, and is what the test pins, because a future "simplification" to copy `textContent` would look right in every screenshot and paste something that is not a path. The copy feedback deliberately flashes on the *icon* button only: `flashCopied()` swaps a button's `innerHTML` for a tick, and the path button's `innerHTML` **is** the path.

### Changed
- **The workspace config is now `skills-config.json` — the project's own committed config, not a format of ours.** Keyed `happyskillsai/instant-canvas`, holding the workspace default `theme` and the `palettes` library. The bespoke dotfile it replaces is **deleted outright** — no migration shim, no back-compat read; it existed only inside this unreleased cycle and nothing external depends on it.
  - **Reads are direct, never a subprocess.** A theme resolves on every canvas load and every hot reload; spawning `npx` per request is not an option. The file is read by HappySkills' documented resolution order (nearest project root, then `~/.agents/`), which is a *supported contract*, not an undocumented fallback.
  - **ABSENT ≠ CORRUPT**, and this is the bug the dotfile shipped with, stated as a rule. A *missing* config means "nothing configured" → defaults, silently — the normal case for a tool launched by `npx` from an arbitrary folder. A config that *exists but does not parse* means the user's settings are unreadable, and calling that "nothing configured" is a **silent failure**. It now throws (`CONFIG_UNREADABLE`), naming the file and the command that will find the defect. It is **never repaired by deleting**: the file holds *every* skill's settings, and a write against a broken one refuses rather than clobbering a neighbour's block.
  - **Writes go through the HappySkills CLI, with an atomic fallback.** `--root <workspace>` is load-bearing — InstantCanvas is launched from any directory, frequently not a HappySkills project at all — and it creates the file if absent. When the CLI is unreachable (offline, cold npx cache) we write it ourselves: atomic, and scoped to our own key so every other skill's block survives. A local-first tool must not fail to save a colour because the user is on a plane.
  - **The `config` schema in `skill.json` is generated from `lib/theme.js`**, not hand-typed. HappySkills enforces the schema we declare, which is strictly good — a bad write is refused at the boundary with the exact path and a fix — but two hand-maintained validators *will* diverge, and a test asserts the shipped `skill.json` and the generator cannot drift.

### Fixed
- **⚠️ A saved palette would have stopped matching its own chip, and nothing would have said why.** `skills-config set` round-trips values exactly — but returns object keys **alphabetised**. The picker matched the active custom-palette chip with `JSON.stringify(a) === JSON.stringify(b)`, which *is* order-sensitive, so the first time a palette went through the CLI its chip would have gone dark while the document was still wearing exactly those colors. Comparison is now **canonical** (keys sorted; arrays keep their order, because a colorway is a sequence), pinned by a test that round-trips a deliberately non-alphabetical palette through the real CLI. The general shape is worth keeping: **any value that survives a round trip through someone else's serializer may come back re-ordered** — if you compare it by string, you have a latent bug whose trigger is "somebody saved it once".
- **Making a palette was unfindable, and the panel was two jobs fighting over one screen.** It is now two screens, and a round **+** in the header is the whole answer to "how do I add my own colors?" — it swaps the body for a real editor (every token, the chart colorway, a name, one Save), with a back arrow out. Three things went wrong before it got there, and each is worth stating because each looked reasonable while it was being written:
  - **Deleting your last palette deleted the feature.** The "Your palettes" section was hidden when the workspace had none — and it also held the name field and Save button, so an empty workspace had no chips, no section, and nothing anywhere saying custom palettes existed. The colour editor that fed it sat *below* the name field, inside a collapsed disclosure labelled "Tokens & colorway", which advertises nothing about saving.
  - **Two buttons read "Save".** One wrote the document, one wrote the workspace library, and they sat a few pixels apart. Naming a palette meant scrolling *past* the thing you were naming, opening a disclosure, tuning, scrolling back up, and then guessing.
  - **Then the first fix reintroduced it.** Keeping the document's footer up in the editor sounded principled — *the footer always means the document* — and rendered as two identical primary buttons both reading "Save", twelve pixels apart. The footer belongs to the list screen. **One screen, one Save.**

  Below the list, pinned above the footer so reading it never costs a scroll, the panel now **says what the selection actually is**: its name, its every token and chart color, and — for the presets — the guidance that until now lived in a `title` tooltip on a chip, where nobody was ever going to meet it. *Colorblind-safe by construction. The only preset that survives a black-and-white printer.* That is the whole reason to choose one of these on grounds other than taste, and it was invisible. The detail also names the state the old panel could not express at all: **a preset with your own colors on top is not that preset any more**, and it now says so ("preset + your changes", plus what you changed and what to do about it) rather than leaving the chip lit and quietly implying otherwise.

  The editor starts from the colors already on screen, because a reader gets there by liking something and wanting to change it, not by wanting a blank slate. If they are looking at one of their own palettes, its **name comes with it**, so saving *updates* that palette (the button says `Update`) instead of silently forking a second near-identical "My brand" — and "edit an existing palette" needs no separate affordance at all. Saving returns to the list, where the chip you just made is sitting there, lit: the new chip is the confirmation, not a toast about a file path.
- **The color picker closed the moment you picked a color, because the panel re-rendered the input the picker was hanging from.** A native `<input type="color">` fires `input` *continuously* while the reader moves around inside the browser's color popup — and that popup is anchored to the input **element**, not to the page. The panel rebuilt its whole token grid on every one of those events, replacing the very node the popup hung from, so the popup shut itself on the first click: you got the color, and lost the picker, every time, with no way to make two adjustments in a row. The panel now separates a **structural** render (the shape changed: a different preset, a swatch added or removed) from a **value sync** that touches no nodes at all — values, active chips, and the reset "×", which is now hidden by an attribute rather than by not existing, precisely so a live edit can reveal it without a rebuild. The input the reader is inside is left alone entirely: writing to its value mid-drag is arguing with the person using it. Same lesson as the detached-node bug below, from the other end — **the DOM is not a pure function of your state when something else is holding a reference into it.**
- **Recoloring during a drag re-rendered every chart on the sheet, per pointer move.** Tokens are a handful of `setProperty` calls and still land on the same frame as the pointer; the Plotly re-render now coalesces to the end of the gesture, which is soon enough to read as live and cheap enough to stay smooth.
- **A container that re-renders on click cannot identify its own clicks after the fact — and the palette panel slammed shut every time you picked a preset.** Its close-on-outside-click handler ran in the **bubble** phase, so by the time it fired, the panel's own handler had already rebuilt the chip grid. The clicked chip was gone from the DOM, `e.target` was a **detached** node, and `e.target.closest('#palettePanel')` on a detached node is `null` — so every click *inside* the panel was indistinguishable from a click outside it. "Was this inside?" is now decided in the **capture** phase, while the target is still in the tree it was clicked in. The bug is invisible in any static reading of the handler, which is why it took driving the real browser to find.
- **A preset shipped to the browser carrying only two of its seven colors, and the CSS fallbacks hid it.** `presetList()` returned the accent and the paper — the two a chip renders — so the page's local preview resolved `text`, `muted`, `border` and `surface` to `undefined`. Nothing looked wrong: `applyDocumentTheme` skips an undefined token, and the sheet's `var(--doc-text, #1a1d24)` fallbacks are exactly the values the preview *should* have produced. It only surfaced downstream, in the two places that read those tokens rather than let CSS resolve them: a saved custom palette lost half its colors, and a chart template compiled Solarized's slate axis ink back to the default near-black. Every token now ships, and a test walks all seven for all fourteen presets. A partial preset lies quietly.
- **`--doc-c1` … `--doc-c8` were written on every themed document and read by no CSS rule anywhere, and are gone.** `applyDocumentTheme` had been setting all eight since document mode shipped; a grep of `styles.css`, `index.html` and `app.js` finds zero readers. They were forward-looking hooks that the future never arrived for: a colorway is a *series* palette, and nothing in the DOM chrome of a sheet is colored by series — the colorway's only real sink is the compiled Plotly template, precisely because Plotly paints to canvas/SVG and cannot read `var()` at all. Deleted rather than wired to an invented rule. What the sheet *does* now take from the theme is the seven-token set: `.sheet`'s hardcoded LIGHT tokens are wired to their `--doc-*` counterparts with the old literals as fallbacks, so a document that declares no theme still renders byte-identically to before. The semantic colors — green/amber/red, and the whole syntax palette — are deliberately *not* themeable: they mean something, and they are not the author's to rebrand.

### Security
- **The palette control is a browser writing to a file, so it is a trust boundary, and it is treated as one.** Theme colors are assigned into live CSS through CSSOM — which the CSP permits precisely because it drops `style=""` attributes, and which would happily accept a string like `javascript:alert(1)`. The canvas validator has always held `document.theme` to strict hex for that reason; `POST /api/theme` now re-checks every token and every colorway entry with the same rules and **refuses rather than sanitizes** (`INVALID_THEME`, 400), because this is the door where a value the validator never saw is about to be persisted into a file the agent will later read back as truth. A test asserts the two gates agree — anything the write boundary accepts must still pass `validate`, or the browser could write a canvas that no longer opens. The route reads nothing that is not already a canvas or a markdown file, so it is not a second way to reach `.env`.
- **A theme is spliced into a canvas as text, never re-serialized.** `lib/jsonedit.js` walks the JSON grammar to find the exact span of `document`'s value and then `theme`'s inside it — a regex cannot, because `"theme"` also occurs inside strings, inside block data, and inside other objects, and rewriting the wrong one would corrupt a user's canvas silently. It adopts the file's own indent and colon spacing, and proves the result by re-parsing and diffing before returning it: a splice that changed anything but `document.theme` is discarded rather than trusted, and the caller re-serializes rather than write something it cannot prove. The same lesson `stamp` learned one level shallower — re-serializing a file the user owns turns a one-line change into a whole-file diff.

## [0.4.0] - 2026-07-12

### Security
- **A file the runtime refused still leaked its own first bytes.** V8's `JSON.parse` error quotes the text it choked on, so any surface that read a path and reported the parse failure was an exfiltration channel: `validate .env` printed `Unexpected token 'D', "DB_PASSWOR"...` onto **stdout — the agent's context**, the one place this project exists to keep secrets out of, and `GET /api/canvas?path=.env` returned the same bytes in a 422. Redaction was no defense (it recognises `sk-`/`AKIA`/`ghp_` shapes, not `DB_PASSWORD`), and neither was confinement (`.env` is *inside* the workspace root — the same blind spot that let `src: ".env"` render in 0.3.0). Both surfaces now decide from the **extension** and never open the file: `assertReadable()` gates every CLI command that takes a path, and `loadCanvas` serves only `*.json` and markdown. Regression-tested across `open`/`print`/`stamp`/`validate` against `.env`, `.txt` and `.yaml`, asserting no byte of the file reaches stdout or stderr.

### Added
- **Markdown files are first-class canvases — an agent no longer writes a wrapper to show one.** `.md` / `.mdx` / `.markdown` files appear in the sidebar on their own, and `open report.md` / `print report.md --out report.pdf` work with no canvas JSON anywhere. The kernel synthesises the envelope in memory (`lib/mdcanvas.js`) and writes nothing to disk, so the whole existing pipeline — image inlining, the paper deck and its auto-TOC, `print`, hot reload, sidebar search — applies unchanged. `print` needs no `document` object on a markdown file: it derives its own paper defaults.
- The workspace scan lists two kinds (`kind: "canvas" | "document"`) and counts them apart (`count`, `docCount`). Documents wear a distinct icon, sort after canvases, and are **never deleted**: collection-delete removes marker-verified canvases only, so a folder holding nothing else is now offered no delete button — a confirmation's count is a promise, and it must equal what the delete performs. The folder browser counts the two apart for the same reason: a folder of markdown is worth opening, but a badge that says "canvases" has to mean canvases.
- A natively-opened markdown file shows only its path in the canvas head. Its title is lifted from its own first heading, which the document is about to render anyway — printing it as chrome as well rendered it twice.
- **The catalog teaches the markdown rule, so it holds for an agent that never reads SKILL.md.** A rule stated only in prose is a rule an agent can compact away — and the one that matters most here is negative: *do not* author a canvas around a `.md` you could have opened. It now lives on the deterministic surface. The lean index opens with `markdownFiles`; it is the **first** note of `catalog markdown`; and `catalog document` states that a `document` object is needed only to print a *canvas* (a `.md` derives its own paper) and that fences wrap rather than scroll on it. The lean index stays inside its cap (which the catalog fix below raises to 7.5 KB, for a reason worth reading). The skill's `description` and keywords also name markdown and PDF, because the trigger vocabulary listed only charts, tables, KPIs, forms and secrets — so "show me this README" or "turn this doc into a PDF" would not have reached InstantCanvas at all.
- **The running header and footer are the reader's, not only the author's.** They used to exist solely if a canvas declared `document.header` / `document.footer`, which meant nobody could add page numbers to a document they had not authored — the common case, now that any markdown file opens as paper. A topbar button (`#stripsBtn`) derives them instead: the canvas title in the header, `{{pageNumber}} / {{totalPages}}` in the footer, on a canvas that declares neither — and the same button drops a *declared* pair, because it is the reader's paper. Unlike the auto-TOC, the strips are not free: `packFragments` probes its per-sheet budget from a sheet **with them attached**, so they take real content height, and turning them on can add a sheet and move every page number after it — the ones printed in the TOC included. The deck is therefore repacked from scratch rather than patched, which is what keeps the TOC honest. The choice lives in memory and resets per canvas, so Cmd+P honours it while `print` renders the declared JSON.

### Fixed
- **SKILL.md could not get an agent to a printable canvas.** `print <canvas.json>` refuses a canvas with no envelope-level `document` object — and SKILL.md's envelope spec omitted that key, never surfaced `catalog document`, and said nothing about what a document carries. An agent following the contract end to end would write a canvas, run `print`, and be refused, with nothing in the skill explaining why. It now carries a Printing section (the `document` shape, `DOCUMENT_INTERACTIVE_BLOCK`, `{{pageNumber}}`, strict-hex theme) and states the one thing an agent cannot discover: a reader can toggle a header/footer on in their browser, but `print` never sees that choice — **declare `footer` if the PDF you generate must carry page numbers**. Also corrected: it advertised bare `catalog` as returning "schemas: all" (it returns none, contradicting its own step 1); `--out` is confined to the workspace, so printing to `/tmp` fails; `validate` takes `--workspace`, without which a markdown `src` resolves against the wrong directory; `--retrofit` permanently writes `"unknown"` and must never be used on a fresh canvas; a sweep needs **two** frames; an authored `markdown` `src` pointing at a README with a badge is a hard `REMOTE_ASSET_BLOCKED` error, not the lenient degrade a natively-opened `.md` gets; `CHROME_PATH` overrides Chrome discovery; and a usage error prints to stderr with **empty stdout**, so "parse the single JSON document on stdout" needs an exit-code check first. Three tests now hold SKILL.md to the CLI — every `document` key it shows must exist in the registry, every `catalog <name>` it offers must resolve, every flag it names must be one the CLI accepts.
- **The catalog — the agent's whole contract — was lying in four places, and an audit found them because nothing tests prose.** The structural tests have always been green: examples validate, the registry cannot drift from the validator. None of that holds a *sentence* to the code, so the teaching text rotted invisibly. (a) **`catalog --full` did not dump everything** — `document` and `sweep` were reachable only by name, so an agent that pulled the whole contract to learn what exists learned they do not; both are in `--full` now. (b) **The lean index was corrupting itself**: `description.split('.')[0]` is not a sentence splitter, so the chart block reached agents as the single word *"Chart."* and confirm as *"Confirmation card (e."* — bytes "saved" by deleting the teaching. It now takes whole sentences, and the size cap rises 6500 → 7500 to pay for saying true things, with a guard that fails on any fragment, cut abbreviation or unbalanced paren. (c) **Validation rules were advertised as flat keys.** Written flat on a field they are merely unknown properties: the canvas validates, and a `minLength` on a password field silently does not exist. The index now names the `"validation"` wrapper and says exactly that. (d) A running header/footer was still described as underivable in `schema.js` after the catalog note was corrected — one contradiction fixed into two.
- **A weighted `graph` threw its weights away.** `encoding.value` has always been documented as edge weight → line width, and the validator even existence-checks the key against your data — then the renderer drew every edge at width 1. The agent shipped weights, got a green validate, and the data vanished with no error anywhere. Edge weight now drives line width: a Plotly scatter trace carries one width, so weighted edges are drawn in a few width bands (heaviest edge thickest). An **unweighted** graph is byte-identical to before — still exactly `[edges, nodes]` — because `options` merges traces by index, and shifting the node trace would silently redirect every existing patch onto an edge.
- **The whole suite could fail 243 tests at once, blaming a file that was fine.** Two root-level `test.before` hooks proved their kernel was up by polling `registry.readAlive()` — which confirms liveness with a 500 ms health ping and **deletes the entry when that ping times out**. That is correct in production (it is what makes `kill -9` recovery automatic) and a trap in a test: under full-suite load the ping loses its race, a perfectly healthy kernel is unregistered, and the hook reports *"kernel did not come up"*. Because `scripts/test/index.js` loads every file into one process, a top-level `before` is a hook on the **root** suite, so one throw took all 243 tests with it — and named the wrong file. The hooks now read the registry raw and confirm liveness themselves. (Test-only; no runtime change.)
- **Document mode had no vertical rhythm at all: every heading was glued to the prose around it.** The deck renders each markdown element as its own `.md.doc-frag`, which makes every element simultaneously `.md > :first-child` *and* `.md > :last-child` — so the margin reset written for the continuous view (where one `.md` holds a whole document) zeroed the top *and* bottom margin of every element on paper. The same document looked perfect on screen, which is why it went unnoticed. Paper typography is now its own layer scoped to `.sheet`, loosened for reading (line-height 1.7, generous heading margins). The margins live inside `flow-root` fragments, so the packer counts them and pagination follows.
- **A code line wider than the page was silently cut off in the PDF.** `overflow: auto` scrolls on screen but *clips* in print — there is nowhere to scroll to on paper, and the reader never learns the line was truncated. Fences now **wrap** on paper (`white-space: pre-wrap; overflow-wrap: anywhere` — `anywhere` rather than `break-word`, so a URL or base64 blob with no break opportunity in it wraps too). Wrapping makes a fence taller and repaginates the deck; that is safe because the packer's hidden measuring replica *is* a `.sheet`, so it measures the same wrapped layout the printer produces. The sheets still equal the PDF's pages 1:1.
- **A wide table lost whole COLUMNS in the PDF, not just their tails.** The same clip, with a far worse payload: the handbook's eleven-column table printed with seven and a half, and `ws_clients`, `idle_seconds` and `version` were absent from the document entirely — no ellipsis, no marker, nothing to tell the reader they had ever existed. On paper a table that would overflow now **folds** instead: `tagWideTables()` measures each one in a real sheet body before packing and tags the offenders, which switches them to `table-layout: fixed` with `break-word` cells. Fixed layout is the only thing that holds — `max-width` alone is quietly ignored (a column's min-content width is a floor the table overflows rather than go under), and dropping that floor with `overflow-wrap: anywhere` stops the clip only by starving every column to one character, printing `id` as "i/d" and `48213` as "48/21/3". Only the offenders fold, because fixed layout divides the page evenly and would stretch a tidy four-column table across the sheet. A cramped column is a cosmetic complaint; a missing column is a lie. Now asserted against the PDF's own text layer — the check that was missing, and the only one that can see this: a DOM assertion passes while the printer still clips.
- **A page of code fences printed a silent sliver page, because the packer measured a DOM the browser was about to grow.** `mountCodeCopy()` ran only *after* the deck was assembled, wrapping each `<pre>` in a `.code-block` — and the wrapper carries a padding of its own, so a wrapped fence is taller than the bare one that was measured. A five-fence sheet came out **160 px over budget**, overflowing with no error anywhere, and the suite stayed green because the deck it was caught on had a small enough budget to hide it; it only surfaced once a reader could enlarge the budget by toggling the running strips off. The wrapper is now mounted over the fragments **before** `packFragments` measures them. The general rule, which outlives the fence: any DOM mutation that can change a fragment's height happens before the packer sees it — measuring a DOM you are about to grow fails exactly as silently as measuring a stale one. (Moving the wrap earlier put it in front of `cloneChain`, so the continuation half of a fence split across two sheets inherits a wrapper with no button in it; `mountCodeCopy()` therefore repairs a buttonless wrapper instead of skipping anything already wrapped.)

### Changed
- **The deck no longer mounts a copy-to-clipboard button** — nobody copies a PDF to the clipboard. `mountCodeCopy(scope, {button: false})` puts the `.code-block` wrapper on (a fence's geometry must be settled before the packer measures it) and no button, so paper holds none at all rather than hiding one at print time. The continuous view keeps one per fence, and the fence on paper reclaims the right padding the button reserved.
- **A markdown file opened natively renders differently than the same file behind an authored `src`, on purpose.** Behind an agent's `src` the validator is a teacher — raw HTML warns, a remote image is a hard `REMOTE_ASSET_BLOCKED` — and the agent fixes the file. A README has no such author, and `html: false` *escapes* rather than deletes, so leaving it alone would print `<details>` as literal text and break every badge. The native view therefore degrades: HTML is removed and its prose kept, an HTML `<img>` becomes a markdown image so a README's logo survives, and a remote image becomes `*(remote image not shown)*`. Fenced code is untouched — a ```` ```html ```` sample is prose *about* markup. `validate` and `stamp` refuse a markdown file: no contract to check, nothing on disk whose birth version could be recorded.

### Security
- **A `markdown` block's `src` accepted any workspace file and rendered it**, so `{"type":"markdown","src":".env"}` displayed the workspace's secrets in the browser. The path was confined to the workspace root, but `.env` lives *inside* the root, so `insideRoot()` admitted it. `src` is now restricted to a `.md`/`.mdx`/`.markdown` extension allowlist (case-insensitive), enforced on **both** surfaces that can read the file: `lib/validate.js`, which the CLI runs, and `lib/kernel.js`, which a canvas can reach without ever passing through the CLI. This deletes SSRF and phone-home at the source: the download happens once, at authoring time, by the agent, outside this software entirely. The error names the URL and its line, and tells the agent to bring the asset local. The kernel's only outbound request remains its own `127.0.0.1/healthz`.

Both are warnings, not errors, because the prose around them still renders. `unsafe-eval` is never granted and the kernel never executes workspace-derived JavaScript.
- **GFM task lists**, via a markdown-it core rule rather than another vendored plugin.
- `scripts/test/markdownsrc.test.js` — the `src` allowlist and the kernel-side read guard, frontmatter stripping, and `data:`-URI image inlining with its confinement, size-cap and fenced-code exclusions. `render.test.js` grows a markdown block asserting no surviving `style=""` attribute, highlighted tokens, an inlined PNG *and* SVG, and a real clipboard round-trip driven over CDP.
- **Canvas search.** A magnifier beside the workspace `+` — or `⌘K`/`Ctrl-K`, or `/` — opens a frosted-glass modal that filters the workspace as you type, matching canvas titles, file names, and the folder holding them, with the matched terms highlighted. Ranked by a title boost, so a name match floats above a folder-only one; interactive canvases are badged. `↑`/`↓` move the selection (wrapping), `↵` opens, `Esc`/backdrop closes and restores focus. **It fetches nothing**: the index is the workspace tree the sidebar already holds, flattened and memoized against the tree object, and the kernel's existing `workspace` WebSocket push keeps it fresh. No new route, no build step, no dependency.
- **The topbar brand mark is an inlined, themeable SVG.** An `<img src="logo.svg">` loads under `img-src 'self'` but renders in an isolated document that cannot see the page's custom properties, so it could never follow the theme. The mark is inlined into `index.html` (like the Lucide icons) and its two fills read `--logo-base`/`--logo-accent`; `assets/logo.svg` remains the canonical standalone file.
- **Two browser interaction tests** (`scripts/test/browse.test.js`, `scripts/test/search.test.js`) driving real headless Chrome through the existing CDP client — they *click*, and assert on what the click did. Both exist because the server was never wrong: `POST /api/browse` returned correct listings while the folder browser was unnavigable, and search has no server side at all. `helpers/cdp.js` now also hands callers the raw `send(method, params)` channel, so `Page.captureScreenshot` and `Emulation.setDeviceMetricsOverride` are reachable.
- **Parameter sweeps.** Any chart kind takes a `sweep` instead of `data`: `{"sweep":{"label"?,"frames":[{"label","data"}]}}`. The agent precomputes every frame; a slider under the chart steps through them. No new block type, no session, no kernel round-trip — a sweep is a property of a display block, so the one-interactive-block rule is untouched. `catalog sweep` returns its schema. Sliders select among precomputed states; live recomputation would need an expression language, and evaluating one needs `unsafe-eval`, which the CSP does not grant.
- **A browser render smoke test** (`scripts/test/render.test.js`) with a zero-dependency CDP client (`scripts/test/helpers/cdp.js`). It renders an adversarial canvas — splom beside violin, a 2-dimension splom, 3D, the skill-rendered kinds, and a sweep — and asserts every chart box drew an SVG root, with zero CSP violations. It catches the class of bug where a chart vanishes silently. Skips when Chrome is absent.
- **9 scientific/ML chart kinds** (26 total): `scatter3d`, `surface`, `contour`, `density`, `violin`, `errorBars`, `dendrogram`, `silhouette`, `splom` — each with its own encoding schema, when-to-use guidance, and a validated example in the registry.
  - `violin` and `errorBars` were structurally impossible under ECharts: both required a JS `renderItem` callback, which the JSON contract forbids.
  - `surface`/`contour` take long-format `{x, y, z}` rows and pivot to a grid. `errorBars` supports `band: true` for learning curves. `dendrogram` consumes a scipy-style linkage where `left`/`right` hold a leaf label or `"#i"` referencing merge `i`; the renderer derives leaf order and bracket geometry. `silhouette` sorts within each cluster, gaps the groups, and draws the mean line.
  - `splom` and `parcoords` are regl-backed and render cleanly under `script-src 'self'` **only because the bundle is built `--strict`**.
- `scatter3d`, `surface` and `splom` mount at 460 px (`.chart-box.tall`) instead of the 320 px default.
- `scripts/web/csp-shim.js` — reconciles Plotly with `style-src 'self'` using Plotly's own `.no-inline-styles` escape hatch, plus a CSSOM reroute for the colorbar's `setAttribute('style', …)`. Zero CSP violations, zero injected `<style>` elements.
- Vendored `plotly.css` (extracted from `build/plotcss.js`) served as a real `'self'` stylesheet.

### Fixed
- **Aligned markdown tables were never aligned, and every one logged a CSP violation.** markdown-it renders a `|---:|` column as `style="text-align:right"`, and `style-src 'self'` drops style *attributes* without an error anyone notices. The alignment now arrives as a class, rewritten from the token before it reaches the DOM. The lesson generalises, and is recorded in the gotchas: any third-party HTML generator is a CSP suspect here, because the failure mode is silence.
- **Inlined SVG images never rendered.** markdown-it's default `validateLink` rejects every `data:` URI except png, jpeg, gif and webp, so the kernel inlined `data:image/svg+xml` correctly and markdown-it discarded the result as literal text — no `<img>`, no error, no CSP violation. Four of the nine supported image types (SVG, AVIF, BMP, ICO) were dead on arrival, and an SVG diagram is the common case. The link validator now accepts exactly the base64 image types the kernel emits; `javascript:`, `vbscript:` and `file:` stay refused.
- **The MDX and raw-HTML warnings misdescribed what happens.** They said the offending lines "will be dropped." They are not: `html:false` *escapes* rather than deletes, so an unremoved `<table>` or `import` line appears verbatim in the rendered document. A teaching warning that gets the outcome wrong sends the agent to the wrong fix. Both now say the lines appear as literal text, and tell the agent to delete them.
- **The folder browser listed the workspace's subfolders but could not be navigated.** Selecting a row called `draw()`, which re-listed the whole `.fb-list` and destroyed the row mid-gesture; since `dblclick` only fires on the common ancestor of both clicks' targets, the second click landed on a freshly created element and never delivered `dblclick` to any row. Double-click was the only way down, so there was none — the modal simply refused to go anywhere, with no error. Selection is now a class toggle that never re-lists, and descending has its own single-click affordance (a per-row chevron) alongside double-click, clickable breadcrumb segments, and `..`. Listing is committed only after the kernel confirms, so an unreadable folder leaves the breadcrumb where it was.
- Closing the search modal opened by `⌘K` or `/` restored focus to `<body>`, stranding keyboard users at the top of the document — `document.activeElement` is the body on those paths, not the trigger. Focus now falls back to the trigger element.
- The `(no subfolders)` message never displayed when a `..` row was present, because the empty-list fallback was short-circuited by string concatenation.
- **`splom` with exactly two dimensions rendered an empty div** — hiding both the diagonal and the upper half left Plotly no cells to draw, with no error and the `.js-plotly-plot` class still applied. The diagonal is kept below three dimensions. This bug also took a neighbouring `violin` down with it (`makeCalcdata of undefined`), so a canvas silently came up one chart short.
- **Charts mount sequentially**, and `rethemeCharts()` serializes its `react` calls. This gives deterministic mount order and lets a `try`/`catch` contain a failing chart. (Note: after fixing `splom`, concurrent mounting alone does not reproduce the dropped-violin failure — the earlier claim that `Plotly.newPlot` is not re-entrant is unproven.)
- The horizontal legend sat on top of the x-axis title (visible on `scatter`, and on every new kind that labels its axes). Legend offset and bottom margin now account for a titled axis.
- Unstacked `area` fills were opaque, burying the series behind them; sankey links used the border colour and were near-invisible. Both now use explicit alpha (`withAlpha()`), with sankey ribbons tinted by source node.

### Notes
- **The npm migration unblocks publishing.** Measured against `happyskills@1.20.1` (`MAX_FILE_SIZE` 1 MB, `MAX_TOTAL_SIZE` 2 MB), the old skill bundle was ~4.4 MB with two single files over the per-file cap (`plotly.min.js` ~2.64 MB, `highlight.min.js` ~1.01 MB). Those files now ship in the npm package instead, and the skill folder is far under both caps. Do not "fix" size by shrinking a vendored bundle — both builds are load-bearing (strict Plotly, class-emitting highlight.js).
- The npm tarball excludes `scripts/test/` via the `files` allowlist; the skill bundle no longer carries the runtime at all.
- The suite is 215 tests: forty-eight drive real headless Chrome (or headless printing) and skip cleanly when it is absent; three exercise the packed npm artifact and skip without npm. `scripts/instantcanvas.js` holds 100% line coverage, enforced by `npm run coverage:cli` (100/77/96 lines/branches/functions).
- **`createdWith` becoming required is the breaking change in this release.** Cutting it needs at least a minor bump, and the release notes must point existing users at `stamp --retrofit`. The stamp value itself is read from `package.json`, so it takes whatever version the release carries — no separate constant to remember to bump.
- The Plotly bundle **must** be built `--strict` (regl-backed `splom`/`parcoords` otherwise call the `Function` constructor and die under `script-src 'self'`) and without map traces (maplibre adds a `blob:` Worker and remote tile hosts). The highlight.js bundle must remain the assembled full build. See `scripts/web/vendor/VENDORED.md` for both recipes.
- New error codes: `MISSING_SOURCE`, `REMOTE_ASSET_BLOCKED`, `MISSING_CREATED_WITH`, `INVALID_CREATED_WITH` — the last two are errors for the CLI and warnings for the kernel. New warning codes: `MDX_NOT_RENDERED`, `RAW_HTML_NOT_RENDERED`.

## [0.1.0] - 2026-07-08

Initial MVP per `specs/260708-01-instantcanvas-mvp`.

### Added
- Canvas contract: envelope (`"instantcanvas": 1`, `blocks` XOR `pages`), 6 block types (`markdown`, `kpi`, `chart` line/bar/pie+donut, `table`, `form`, `confirm`), 16 form field types; declarative schema registry driving both the validator and `catalog`.
- Deterministic validator: all errors in one pass with `code`/`path`/`message`, Levenshtein + alias "Did you mean" hints, examples; unknown properties as warnings.
- CLI (`open` / `validate` / `catalog` / `status` / `stop`): stdout = exactly one JSON document, redacted stderr logs, exit 0/1/2 contract; display canvases return immediately, interactive canvases block on the human.
- Per-workspace kernel: 127.0.0.1-only, per-kernel token (timing-safe), Host-header check, CSP, hand-rolled RFC 6455 WebSocket hot reload, idle auto-shutdown, health-ping registry with stale-entry and kill -9 recovery.
- Secure forms: values written to `.env` (parse-preserving merge) or JSON files; overwrite and outside-workspace writes require in-browser confirmation; secrets registered with the redaction layer before any processing and excluded from every result, log, and error.
- Frontend: prototype-faithful shell (light/dark), vendored ECharts 5.6.0 + markdown-it 14.3.0 (served, never required), hot-reload client, folder browser.
