# Project spec rules — instant-canvas

These rules are applied by `init-spec` (authoring mode) **on top of** its generic
spec-authoring rules. Every `SPEC.md` written in this project must obey them, and the
load-bearing ones must be **embedded into the spec itself** (§0, §3 Acceptance, §5
Non-goals, §4 tasks) so the fresh implementer session honors them without reading this file.

---

## 1. Skill-sync assessment is MANDATORY in every spec (the gate)

The agent never drives the CLI directly — it drives the **skill**
(`.agents/skills/instant-canvas/SKILL.md` + `skill.json`), and the skill drives the CLI.
A CLI capability the skill does not describe is **invisible to the agent**, no matter how well
it works. Every spec MUST contain an explicit, up-front section (in §0 or its own numbered
section) that answers, out loud:

> **Does this change require updating the agent-facing skill?** (yes / no — and why.)

A spec that leaves this implicit is incomplete.

The answer is **YES** whenever the change **adds, renames, removes, or alters** any of: a CLI
command, a flag, a stdout/output field or shape, an exit code, an error code, or a user-facing
concept the agent reasons about — **or** changes or deletes existing CLI behavior.

The answer is **NO** only when the agent **cannot observe the change through the CLI contract**:
web-app UI / look-and-feel, a CSS re-theme, a browser-only interaction, or an internal refactor
with no change to commands, flags, output, exit codes, or behavior.

---

## 2. If YES → the spec must SPECIFY HOW to update the skill, researched and verified

Updating the skill correctly is **not mechanical** — this is the rule's whole reason to exist.
"Yes, update the skill" is a failing answer. The spec MUST include a **dedicated §4 task** whose
instructions carry the following, concretely, so a fresh implementer cannot hand-wave it:

**2a. Research the REAL surface first — never document from memory or from the code alone.**
The task MUST instruct the implementer to (i) read the CLI change to enumerate exactly what is
new/changed (command, flags, output fields, exit/error codes, behavior); (ii) **drive the running
CLI and capture its ACTUAL stdout**, then transcribe the output contract from what the CLI really
emits — verified field-for-field, including the empty/edge case (e.g. `count: 0`, a `null`
timestamp) and any optional field (e.g. `dropped?`); and (iii) read how `SKILL.md` already teaches
a comparable command and **mirror that structure and precision** (house style), not invent a new
shape.

**2b. Update BOTH halves of the contract — they are separate mechanisms.**
- **The body (how to drive it):** the prose section teaching the concept and the workflow (what
  the capability is *for*, not just its syntax), the `## Commands` block entry (command + every
  flag), and the **exact output contract** in the result-handling section (every field, the empty
  case, optional fields, exit codes, the one-JSON-document-on-stdout discipline).
- **The auto-trigger (when to reach for it):** the `description` in **both** the `SKILL.md`
  frontmatter **and** `skill.json`. This is a DISTINCT mechanism from the body — a body-only update
  leaves the skill fully documented but **un-triggerable** for the new capability, so the agent
  never loads it. The task MUST call out the description update explicitly whenever the change
  introduces a new user-facing concept or trigger vocabulary.

**2c. Respect the description constraints (they are strict and easy to violate).**
The `SKILL.md` frontmatter `description` has a **≤250-char cap** (target 80–180), forbids the YAML
characters `; : # " ' [ ] ! & * % | >` **even inside quotes** (use an em-dash, never a colon), and
must pack the trigger vocabulary into a single sentence; `skill.json`'s `description` has its own
~200-char guide. When near the cap, **trim existing wording to make room** rather than exceed it.
The task MUST note these limits so the implementer does not break the publish validators.

**2d. Never edit the skill bundle's `CHANGELOG.md`.** It is owned by the publish step. A feature
spec's skill changes go in `SKILL.md` / `skill.json` only.

---

## 3. If YES → §3 must carry a VERIFIABLE skill↔CLI parity acceptance criterion

Not "the skill was updated" — a check a fresh session can run. The spec MUST embed the
deterministic reconciliation audit this project uses:

> Enumerate the CLI surface from source — `grep` the command dispatch (`case '<cmd>'`), the flags
> `parseArgs` accepts (`'--<flag>'`), and every `out({ status: '<...>' })` the CLI emits — and
> assert that **every** command, flag, and `status` string is represented in `SKILL.md` with its
> **exact** output shape, and that the `description` (frontmatter **and** `skill.json`) carries the
> new trigger vocabulary. The output shapes must match the **running** CLI, not the source's
> object literals — drive it and diff.

A spec whose acceptance criteria a reviewer could pass while the skill still lags the CLI has not
satisfied this rule.

---

## 4. If NO → state the exemption and its reason (never a silent omission)

When the assessment is NO, the spec MUST say so in one line with the reason (web-app UI, CSS
re-theme, browser-only interaction, internal refactor), so the omission is a recorded decision.
